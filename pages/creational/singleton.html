1️⃣ Singleton Pattern

Ensures a class has only one instance and provides a global point of access to it. There are 6 different ways to implement Singleton in Java — each with different trade-offs in thread safety, performance, and robustness.

⚡ Quick Jump — All 6 Approaches

1. Eager Init 2. Lazy Init 3. Synchronized Method 4. Double-Checked Locking 5. Bill Pugh ⭐ Best 6. Enum Singleton

🎯 When to Use Singleton

  • Database connection pools (HikariDataSource)
  • Logger instances (Logger.getLogger())
  • Configuration / Properties managers
  • Thread pools, Cache managers
  • Spring beans (default scope is singleton — managed by Spring container)
  • Application-wide shared state (e.g., counter, registry)

🔵 Approach 1 — Eager Initialization

Instance is created at class-loading time. Simplest approach. Thread-safe by JVM class-loading guarantee. Drawback: object is created even if never used.

EagerSingleton.java 
public class EagerSingleton {

    // Instance created when class loads — before getInstance() is ever called
    private static final EagerSingleton INSTANCE = new EagerSingleton();

    // Private constructor — no one can call new EagerSingleton()
    private EagerSingleton() {
        System.out.println("EagerSingleton created at class load time");
    }

    public static EagerSingleton getInstance() {
        return INSTANCE; // Always returns the same pre-created instance
    }

    public void doWork() {
        System.out.println("Working... hashCode: " + this.hashCode());
    }

    // Usage
    public static void main(String[] args) {
        EagerSingleton s1 = EagerSingleton.getInstance();
        EagerSingleton s2 = EagerSingleton.getInstance();
        System.out.println(s1 == s2);            // true — same object
        System.out.println(s1.hashCode() == s2.hashCode()); // true
    }
}
PropertyValue
Thread Safe✅ Yes (JVM class loading)
Lazy Loading❌ No — created even if unused
Exception Handling❌ Cannot handle exceptions in constructor
Use WhenObject is always needed + creation is cheap

🟡 Approach 2 — Lazy Initialization (NOT Thread-Safe)

Instance is created only when first requested. ⚠️ Not safe for multi-threaded environments — two threads can create two instances simultaneously.

LazySingleton.java ⚠️ Not Thread-Safe 
public class LazySingleton {

    private static LazySingleton instance; // null initially

    private LazySingleton() {
        System.out.println("LazySingleton created");
    }

    public static LazySingleton getInstance() {
        // ⚠️ PROBLEM: Two threads can both see instance == null simultaneously
        // Thread A checks → null, Thread B checks → null
        // Both create a new instance → TWO instances exist!
        if (instance == null) {
            instance = new LazySingleton(); // Race condition here!
        }
        return instance;
    }
}

// ⚠️ Thread-unsafe scenario:
// Thread A: if (instance == null) → true
// Thread B: if (instance == null) → true  ← both pass check!
// Thread A: instance = new LazySingleton() ← creates object 1
// Thread B: instance = new LazySingleton() ← creates object 2 ← BROKEN!
⚠️

Only use this in single-threaded applications. In a Spring Boot or any multi-threaded Java app, this will create multiple instances and break the Singleton guarantee.

🟠 Approach 3 — Thread-Safe (Synchronized Method)

Adds synchronized to getInstance() — only one thread can execute it at a time. Thread-safe but performance bottleneck: every call to getInstance() acquires a lock even after instance is created.

SynchronizedSingleton.java 🐢 Thread-Safe but Slow 
public class SynchronizedSingleton {

    private static SynchronizedSingleton instance;

    private SynchronizedSingleton() {
        System.out.println("SynchronizedSingleton created");
    }

    // synchronized ensures only ONE thread can enter at a time
    // ✅ Thread-safe — but 🐢 every call acquires the lock (100x slower!)
    public static synchronized SynchronizedSingleton getInstance() {
        if (instance == null) {
            instance = new SynchronizedSingleton();
        }
        return instance;
    }
}

// Problem: After instance is created (99.99% of calls),
// ALL threads STILL wait to acquire the lock unnecessarily.
// In a high-traffic Spring app, this is a performance bottleneck.
PropertyValue
Thread Safe✅ Yes
Lazy Loading✅ Yes
Performance❌ Poor — every call locks, even post-init
Use WhenSimple low-traffic, single-threaded code. Avoid in production.

🟢 Approach 4 — Double-Checked Locking (DCL)

Fixes the performance problem of Approach 3. Only locks when creating — after instance exists, no lock needed. Requires volatile to prevent instruction reordering by the JVM/CPU.

DCLSingleton.java ✅ Thread-Safe + Performant 
public class DCLSingleton {

    // volatile: prevents CPU instruction reordering during object creation
    // Without volatile, a thread might see a partially constructed object
    private static volatile DCLSingleton instance;

    private DCLSingleton() {
        System.out.println("DCLSingleton created");
    }

    public static DCLSingleton getInstance() {

        // 1st CHECK — no lock needed if already created (99.99% of calls go here)
        if (instance == null) {

            // Lock only when instance might need to be created
            synchronized (DCLSingleton.class) {

                // 2nd CHECK — inside the lock, verify again
                // (another thread might have created it between 1st check and lock)
                if (instance == null) {
                    instance = new DCLSingleton(); // ← only ONE thread ever reaches here
                }
            }
        }
        return instance; // fast path — no lock for subsequent calls
    }
}

// Why volatile is critical:
// Object creation has 3 steps internally:
//   1. Allocate memory
//   2. Initialize the object (constructor)
//   3. Assign reference to 'instance'
//
// CPU can reorder → step 3 before step 2!
// Thread B might see instance != null but object not yet initialized.
// volatile forces "happens-before" relationship — step 3 only after step 2.
PropertyValue
Thread Safe✅ Yes (with volatile)
Lazy Loading✅ Yes
Performance✅ Good — lock only during creation
Complexity⚠️ Easy to get wrong (forget volatile)
Use WhenNeed lazy loading in multi-threaded environment

⭐ Approach 5 — Bill Pugh Singleton (Static Inner Class)

The most recommended approach for most use cases. Invented by Bill Pugh in 2004. Uses a static inner holder class — instance is NOT created until getInstance() is called. JVM class loading guarantee makes it thread-safe with zero synchronization overhead.

💡

Why it works: Java's class loader only loads the inner SingletonHolder class when it's first referenced (inside getInstance()). Class loading is thread-safe by the JVM — no synchronized needed.

BillPughSingleton.java ⭐ RECOMMENDED — Lazy + Thread-Safe + No Lock 
public class BillPughSingleton {

    // Private constructor — prevents external instantiation
    private BillPughSingleton() {
        // Guard against reflection attack
        if (SingletonHolder.INSTANCE != null) {
            throw new IllegalStateException("Use getInstance() — Singleton already created!");
        }
        System.out.println("BillPughSingleton instantiated");
    }

    // ─── Static Inner Holder Class ────────────────────────────
    // This class is NOT loaded when BillPughSingleton is loaded.
    // It is loaded ONLY when getInstance() is called for the first time.
    // JVM class loading is thread-safe → no synchronized keyword needed!
    private static class SingletonHolder {
        private static final BillPughSingleton INSTANCE = new BillPughSingleton();
        //                                                 ↑
        //                    Created HERE — when SingletonHolder is first loaded
        //                    This happens exactly ONCE, automatically thread-safe
    }

    // Public access point
    public static BillPughSingleton getInstance() {
        return SingletonHolder.INSTANCE;
        // ↑ Triggers SingletonHolder class loading (if not loaded already)
        //   After first load, this is just a field read — super fast!
    }

    // Prevent deserialization from creating a new instance
    protected Object readResolve() {
        return getInstance();
    }

    // ─── Your actual business methods ────────────────────────
    public void connect() {
        System.out.println("Connected! Instance hashCode: " + hashCode());
    }
}

// ─── Usage ────────────────────────────────────────────────────
public class Main {
    public static void main(String[] args) throws InterruptedException {
        // Simulate multiple threads calling getInstance() simultaneously
        Runnable task = () -> {
            BillPughSingleton s = BillPughSingleton.getInstance();
            s.connect();
            System.out.println(Thread.currentThread().getName()
                + " → hashCode: " + s.hashCode());
        };

        Thread t1 = new Thread(task, "Thread-1");
        Thread t2 = new Thread(task, "Thread-2");
        Thread t3 = new Thread(task, "Thread-3");

        t1.start(); t2.start(); t3.start();
        t1.join();  t2.join();  t3.join();

        // All 3 threads get the SAME hashCode — proves single instance
    }
}

// OUTPUT (hashCode same for all threads):
// BillPughSingleton instantiated  ← printed only ONCE
// Connected! Instance hashCode: 12345678
// Thread-1 → hashCode: 12345678
// Thread-2 → hashCode: 12345678
// Thread-3 → hashCode: 12345678

Real-World Example — Database Connection Pool with Bill Pugh

ConnectionPool.java — Production-Grade Bill Pugh 
import java.sql.Connection;
import java.sql.DriverManager;
import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.BlockingQueue;

public class ConnectionPool {

    private static final int POOL_SIZE = 10;
    private final BlockingQueue<Connection> pool;
    private final String jdbcUrl;

    // Private constructor — expensive initialization
    private ConnectionPool() {
        this.jdbcUrl = System.getenv().getOrDefault("DB_URL", "jdbc:mysql://localhost:3306/app");
        this.pool = new ArrayBlockingQueue<>(POOL_SIZE);
        try {
            for (int i = 0; i < POOL_SIZE; i++) {
                pool.add(DriverManager.getConnection(jdbcUrl, "root", "secret"));
            }
            System.out.println("Connection pool initialized with " + POOL_SIZE + " connections");
        } catch (Exception e) {
            throw new RuntimeException("Failed to initialize connection pool", e);
        }
    }

    // Bill Pugh Holder — lazy, thread-safe, no lock
    private static class Holder {
        private static final ConnectionPool INSTANCE = new ConnectionPool();
    }

    public static ConnectionPool getInstance() {
        return Holder.INSTANCE;
    }

    public Connection getConnection() throws InterruptedException {
        return pool.take(); // blocks until a connection is available
    }

    public void returnConnection(Connection conn) {
        pool.offer(conn); // return connection to pool
    }

    // Usage
    public static void main(String[] args) throws Exception {
        // First call → creates pool (expensive, once)
        ConnectionPool pool1 = ConnectionPool.getInstance();

        // Second call → returns same pool (instant, no cost)
        ConnectionPool pool2 = ConnectionPool.getInstance();

        System.out.println("Same pool? " + (pool1 == pool2)); // true

        // Use connections
        Connection conn = pool1.getConnection();
        // ... use connection ...
        pool1.returnConnection(conn);
    }
}
PropertyValue
Thread Safe✅ Yes — JVM class loading guarantee
Lazy Loading✅ Yes — inner class not loaded until needed
Performance✅ Excellent — no synchronization overhead
Reflection Safe⚠️ Needs manual guard in constructor
Serialization Safe⚠️ Needs readResolve()
Complexity✅ Simple and clean
Use WhenDefault choice for most Java applications

🔴 Approach 6 — Enum Singleton (Joshua Bloch's Recommendation)

Using a Java enum with a single constant. Most robust — JVM guarantees exactly one instance even against reflection, serialization, and cloning. Recommended in Effective Java by Joshua Bloch as the preferred Singleton approach when you don't need lazy loading.

AppConfig.java — Enum Singleton (Safest) 
// Enum Singleton — most concise, reflection-safe, serialization-safe
public enum AppConfig {
    INSTANCE; // ← Single constant = Single instance

    // Fields
    private final String dbUrl;
    private final String appName;
    private int maxRetries;

    // Enum constructor — called exactly ONCE by JVM
    AppConfig() {
        this.dbUrl      = System.getenv().getOrDefault("DB_URL", "jdbc:h2:mem:testdb");
        this.appName    = "MyApplication";
        this.maxRetries = 3;
        System.out.println("AppConfig enum initialized");
    }

    // Getters
    public String getDbUrl()    { return dbUrl; }
    public String getAppName()  { return appName; }
    public int getMaxRetries()  { return maxRetries; }

    // Setters for mutable config
    public void setMaxRetries(int retries) {
        this.maxRetries = retries;
    }

    // Business method
    public void printConfig() {
        System.out.println("App: " + appName + " | DB: " + dbUrl + " | Retries: " + maxRetries);
    }
}

// Usage — access via INSTANCE constant
public class Application {
    public static void main(String[] args) {
        // Access singleton
        AppConfig config = AppConfig.INSTANCE;
        config.printConfig();

        // Works across all classes
        String url = AppConfig.INSTANCE.getDbUrl();

        // Enum is automatically:
        // ✅ Thread-safe (JVM enum guarantee)
        // ✅ Reflection-safe (JVM blocks reflective construction of enums)
        // ✅ Serialization-safe (enum has built-in serialization)
        // ✅ Clone-safe (enums cannot be cloned)

        // Proof — reflection CANNOT break it:
        try {
            java.lang.reflect.Constructor<AppConfig> c =
                AppConfig.class.getDeclaredConstructor(String.class, int.class);
            c.setAccessible(true);
            c.newInstance("HACK", 0); // ← JVM throws IllegalArgumentException!
        } catch (Exception e) {
            System.out.println("✅ Reflection blocked: " + e.getMessage());
            // "Cannot reflectively create enum objects"
        }
    }
}
PropertyValue
Thread Safe✅ Yes — JVM enum guarantee
Lazy Loading❌ No — created at class load
Reflection Safe✅ Yes — JVM prevents it
Serialization Safe✅ Yes — built into enum
Clone Safe✅ Yes — enum cannot be cloned
Inheritance❌ Cannot extend other classes
Use WhenNeed maximum safety; don't need lazy loading or inheritance

🔓 How to Break Singleton & How to Prevent

BreakSingleton.java — 3 Attacks + Fixes 
// ── Attack 1: Reflection ──────────────────────────────────────
Constructor<BillPughSingleton> c = BillPughSingleton.class.getDeclaredConstructor();
c.setAccessible(true);
BillPughSingleton s2 = c.newInstance(); // creates 2nd instance without guard!

// FIX: In constructor, throw if instance already exists:
private BillPughSingleton() {
    if (SingletonHolder.INSTANCE != null) {
        throw new IllegalStateException("Singleton already exists! Use getInstance()");
    }
}

// ── Attack 2: Serialization ────────────────────────────────────
// Serialize instance → write to file → deserialize → NEW object!
ObjectOutputStream out = new ObjectOutputStream(new FileOutputStream("singleton.ser"));
out.writeObject(BillPughSingleton.getInstance());
ObjectInputStream in = new ObjectInputStream(new FileInputStream("singleton.ser"));
BillPughSingleton deserialized = (BillPughSingleton) in.readObject(); // new instance!

// FIX: Add readResolve() method:
protected Object readResolve() {
    return getInstance(); // always return the existing instance
}

// ── Attack 3: Cloning ──────────────────────────────────────────
BillPughSingleton cloned = (BillPughSingleton) getInstance().clone(); // new instance!

// FIX: Override clone() to throw exception:
@Override
protected Object clone() throws CloneNotSupportedException {
    throw new CloneNotSupportedException("Singleton cannot be cloned!");
}

// ── BEST FIX: Use Enum ─────────────────────────────────────────
// Enum is immune to ALL 3 attacks automatically — JVM guarantees it.

🎤 Interview Q&A

Bill Pugh Singleton uses a static inner holder class to hold the singleton instance. It is the best because:
(1) Lazy loading — inner class not loaded until getInstance() is first called.
(2) Thread-safe — JVM class loading is inherently thread-safe (no synchronized needed).
(3) No performance overhead — no lock, no volatile, just a field read after initialization.
(4) Simple code — cleaner than DCL. Introduced by Bill Pugh in 2004 in the paper "Initialization on Demand Holder".

DCL: Uses volatile + synchronized. Correct if volatile is not forgotten. Programmer-managed thread safety. Slightly complex.
Bill Pugh: Uses JVM's class loading guarantee (static inner class). No volatile, no synchronized. JVM-managed thread safety. Simpler and faster. Preferred over DCL in modern Java.

volatile prevents CPU instruction reordering. Object creation has 3 steps: (1) allocate memory, (2) call constructor, (3) assign reference. Without volatile, the CPU can reorder steps to: (1) allocate, (3) assign reference, (2) call constructor. Thread B checking instance != null would get a non-null reference to an uninitialized object and crash with NPE. volatile enforces the happens-before relationship.

Enum Singleton is the only approach that is 100% reflection-proof by JVM specification. You cannot use reflection to create a new enum instance — the JVM throws IllegalArgumentException: Cannot reflectively create enum objects. All other approaches (Eager, Lazy, DCL, Bill Pugh) can be broken via reflection unless you add a guard check in the constructor.

1. Reflection: Use getDeclaredConstructor().setAccessible(true).newInstance(). Fix: throw in constructor if instance already exists.
2. Serialization: Deserializing a serialized singleton creates a new object. Fix: implement readResolve() returning getInstance().
3. Cloning: clone() on a Cloneable object creates a copy. Fix: override clone() to throw CloneNotSupportedException.
Best fix: Use Enum Singleton — JVM automatically prevents all three.

Singleton Pattern (GoF): One instance per JVM/ClassLoader. Class manages its own lifecycle. Accessed via static getInstance(). Difficult to mock/test. Manual thread safety.
Spring @Scope("singleton"): One instance per Spring ApplicationContext. Spring manages lifecycle (creation, DI, destruction). Injected via DI — testable with mocks. Thread safety is application responsibility. Multiple contexts → multiple instances. Prefer Spring's approach in enterprise apps.

SRP violation: The class is responsible for both its business logic AND for ensuring only one instance is created — two responsibilities.
DIP violation: Classes calling Singleton.getInstance() hard-code their dependency on the concrete singleton — not an abstraction.
Testability issue: Hard to replace with a mock in unit tests.
Mitigation: Use Spring's IoC container — it manages singleton lifecycle externally, keeping your class clean and testable.

📌

Summary: Lazy (unsafe) → Eager → Synchronized (slow) → DCL + volatile → Bill Pugh ⭐ (best for most) → Enum (safest). In Spring Boot? Just use @Component — Spring handles it all.
