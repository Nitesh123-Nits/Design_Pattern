1️⃣ Singleton Pattern

Ensures a class has only one instance and provides a global point of access to it. One of the most commonly asked patterns in Java interviews.

🎯 When to Use

  • Database connection pools
  • Logger instances (e.g., Logger.getLogger())
  • Configuration managers
  • Thread pools
  • Caches
  • Spring beans (default scope is singleton)

💻 Java Implementation

1. Thread-Safe Singleton (Double-Checked Locking) ← Preferred

DatabaseConnectionPool.java
public class DatabaseConnectionPool {

    // volatile ensures visibility across threads — prevents instruction reordering
    private static volatile DatabaseConnectionPool instance;

    private final String url;
    private final int maxConnections;

    // Private constructor — prevents direct instantiation
    private DatabaseConnectionPool() {
        this.url = "jdbc:mysql://localhost:3306/mydb";
        this.maxConnections = 10;
        System.out.println("DatabaseConnectionPool initialized");
    }

    // Double-Checked Locking — thread-safe & performant
    public static DatabaseConnectionPool getInstance() {
        if (instance == null) {                        // 1st check — avoids synchronization overhead
            synchronized (DatabaseConnectionPool.class) {
                if (instance == null) {                // 2nd check — within synchronized block
                    instance = new DatabaseConnectionPool();
                }
            }
        }
        return instance;
    }

    public String getUrl() { return url; }
    public int getMaxConnections() { return maxConnections; }

    // Prevent cloning
    @Override
    protected Object clone() throws CloneNotSupportedException {
        throw new CloneNotSupportedException("Singleton cannot be cloned");
    }

    // Prevent deserialization from creating new instance
    protected Object readResolve() {
        return instance;
    }
}

2. Enum Singleton ← Best for Serialization Safety

AppConfig.java (Enum Singleton)
// Most concise, serialization-safe, reflection-safe Singleton
public enum AppConfig {
    INSTANCE;

    private String dbUrl;
    private int timeout;

    // Enum constructor (called once by JVM)
    AppConfig() {
        this.dbUrl = System.getenv().getOrDefault("DB_URL", "jdbc:h2:mem:testdb");
        this.timeout = 30;
    }

    public String getDbUrl() { return dbUrl; }
    public int getTimeout() { return timeout; }
    public void setDbUrl(String url) { this.dbUrl = url; }

    // Usage:
    // AppConfig.INSTANCE.getDbUrl()
    // AppConfig.INSTANCE.getTimeout()
}

3. Eager Initialization

Logger.java (Eager)
// Good when instantiation is cheap and always needed
public class AppLogger {
    // Created when class is loaded — thread-safe by JVM class loading guarantee
    private static final AppLogger INSTANCE = new AppLogger();

    private AppLogger() {}

    public static AppLogger getInstance() {
        return INSTANCE;
    }

    public void log(String message) {
        System.out.println("[LOG] " + java.time.LocalDateTime.now() + " - " + message);
    }
}

// Usage
public class Main {
    public static void main(String[] args) {
        // Both references point to the SAME object
        DatabaseConnectionPool pool1 = DatabaseConnectionPool.getInstance();
        DatabaseConnectionPool pool2 = DatabaseConnectionPool.getInstance();
        System.out.println(pool1 == pool2); // true

        AppLogger.getInstance().log("Application started");
        AppConfig.INSTANCE.getDbUrl(); // Enum singleton
    }
}

📊 Pros & Cons

AspectDetail
✅ Single instanceControlled access to a single shared resource
✅ Lazy initializationCreated only when needed (saves memory)
✅ Global accessAccessible from anywhere without passing references
❌ Global stateMakes unit testing difficult — hard to mock
❌ Thread safetyRequires careful implementation in multithreaded environments
❌ Violation of SRPClass manages its own lifecycle + business logic
❌ Hidden dependencyClasses calling getInstance() hide their dependency

🔍 Real-World Usage in Java/Spring

  • java.lang.Runtime.getRuntime() — JVM Runtime instance
  • java.awt.Desktop.getDesktop()
  • java.lang.System.console()
  • Spring beans (default scope singleton) — Spring manages the lifecycle
  • HikariCP.HikariDataSource — typically a singleton connection pool

🎤 Interview Q&A

1. Reflection: Constructor c = Singleton.class.getDeclaredConstructor(); c.setAccessible(true); Singleton s2 = c.newInstance(); — Fix: throw exception in constructor if instance already exists.
2. Serialization/Deserialization: deserialization creates a new object. Fix: implement readResolve() to return existing instance.
3. Cloning: clone() creates a new instance. Fix: override clone() to throw CloneNotSupportedException.
Best fix: Use Enum Singleton — JVM guarantees protection against all three automatically.

volatile prevents CPU instruction reordering. Without it, the JVM might reorder: (1) allocate memory, (2) assign reference to instance, (3) call constructor. Thread 2 could see a non-null but uninitialized object. volatile ensures the write to instance is fully visible to all threads only after the constructor completes.

Yes, some consider it an anti-pattern because: global state makes testing difficult, it hides dependencies, and it violates SRP. Prefer Dependency Injection (e.g., Spring singleton beans) over manual Singleton. Use Singleton sparingly — only for truly shared, stateless or carefully managed stateful resources like connection pools, loggers, or configuration.

Singleton Pattern: one instance per JVM/ClassLoader, managed by the class itself, accessed via static method, difficult to test.
Spring Singleton scope: one instance per Spring Application Context/IoC container, managed by Spring, injected via DI, fully testable with mock injection. Spring's is preferred in enterprise apps — it gives you IoC + lifecycle management + testability.

Singleton is per-JVM. In a cluster with multiple JVM instances, each JVM has its own Singleton instance. If you need a truly global single instance across a cluster, use: (1) a distributed cache (Redis, Hazelcast) for shared state, (2) a database lock or leader election (ZooKeeper, etcd), (3) a shared service (microservice that owns the resource). Plain Singleton is not cluster-aware.

📌

Quick Comparison: Eager (simple) → DCL (lazy + thread-safe) → Enum (serialization-safe + simplest). For most Spring apps, just use @Service / @Component beans which are singleton by default.
