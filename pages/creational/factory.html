🏭 Factory Method Pattern

Defines an interface for creating objects, but lets subclasses decide which class to instantiate. Factory Method lets a class defer instantiation to subclasses.

🎯 When to Use

  • When you don't know ahead of time what type of object to create
  • When subclasses should decide which concrete class to instantiate
  • When you want to provide a "hook" for subclasses to extend creation logic
  • When you want to centralize object creation logic

💻 Java Implementation

Notification Factory — Real-World Example
// Product interface
public interface Notification {
    void send(String title, String message);
    String getChannel();
}

// Concrete Products
public class EmailNotification implements Notification {
    private final String email;
    public EmailNotification(String email) { this.email = email; }

    @Override
    public void send(String title, String message) {
        System.out.println("📧 EMAIL to " + email + " | Title: " + title + " | " + message);
    }
    @Override public String getChannel() { return "EMAIL"; }
}

public class SMSNotification implements Notification {
    private final String phone;
    public SMSNotification(String phone) { this.phone = phone; }

    @Override
    public void send(String title, String message) {
        System.out.println("📱 SMS to " + phone + " | " + message);
    }
    @Override public String getChannel() { return "SMS"; }
}

public class PushNotification implements Notification {
    private final String deviceToken;
    public PushNotification(String deviceToken) { this.deviceToken = deviceToken; }

    @Override
    public void send(String title, String message) {
        System.out.println("🔔 PUSH to device " + deviceToken + " | " + title + ": " + message);
    }
    @Override public String getChannel() { return "PUSH"; }
}

// Creator (abstract) — defines the factory method
public abstract class NotificationFactory {
    // Factory Method — subclasses override this
    public abstract Notification createNotification(String recipient);

    // Template method that uses the factory
    public final void notify(String recipient, String title, String message) {
        Notification notification = createNotification(recipient);
        System.out.println("Sending via: " + notification.getChannel());
        notification.send(title, message);
    }
}

// Concrete Creators
public class EmailNotificationFactory extends NotificationFactory {
    @Override
    public Notification createNotification(String recipient) {
        return new EmailNotification(recipient);
    }
}

public class SMSNotificationFactory extends NotificationFactory {
    @Override
    public Notification createNotification(String recipient) {
        return new SMSNotification(recipient);
    }
}

// --- Simple Factory (NOT GoF, but commonly asked) ---
// A static factory: no subclasses, just a switch
public class NotificationFactorySimple {
    public static Notification create(String type, String recipient) {
        return switch (type.toUpperCase()) {
            case "EMAIL" -> new EmailNotification(recipient);
            case "SMS"   -> new SMSNotification(recipient);
            case "PUSH"  -> new PushNotification(recipient);
            default -> throw new IllegalArgumentException("Unknown type: " + type);
        };
    }
}

// Usage
public class Main {
    public static void main(String[] args) {
        // Factory Method pattern
        NotificationFactory factory = new EmailNotificationFactory();
        factory.notify("user@example.com", "Order Confirmed", "Your order #123 is confirmed");

        factory = new SMSNotificationFactory();
        factory.notify("+91-9999999999", "OTP", "Your OTP is 456789");

        // Simple Factory
        Notification n = NotificationFactorySimple.create("PUSH", "device-token-xyz");
        n.send("Sale!", "50% off today only!");
    }
}

📊 Pros & Cons

AspectDetail
✅ Loose couplingClient code depends on Product interface, not concrete classes
✅ Open/ClosedAdd new product types by creating new factory subclass
✅ SRPCreation logic is centralized and separated from business logic
❌ Class proliferationEach new product requires a new factory subclass
❌ ComplexitySimple object creation becomes overly abstract

🎤 Interview Q&A

Simple Factory: A static method that creates objects based on a parameter (not a GoF pattern, but widely used).
Factory Method: A GoF pattern where a class defines a method that returns a product, and subclasses override it to change the product type. The factory IS the class.
Abstract Factory: Creates families of related objects. A factory of factories. Returns multiple related products that are designed to work together.

You can add a new notification type (e.g., WhatsAppNotification) by creating a new WhatsAppNotificationFactory subclass — without modifying any existing code. The abstract creator and other concrete creators remain unchanged. This is exactly OCP: extend by adding new classes, not by modifying existing ones.

Use Factory Method when: (1) The exact type to create isn't known until runtime, (2) Creation involves complex logic (validation, configuration), (3) You want to encapsulate "which class" decisions, (4) You want subclasses to control what gets created (extensibility). Avoid if creation is trivial — overengineering defeats the purpose.
