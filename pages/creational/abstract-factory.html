🔧 Abstract Factory Pattern

Provides an interface for creating families of related or dependent objects without specifying their concrete classes. A "factory of factories."

🎯 When to Use

  • When your system needs to be independent of how its products are created
  • When you need to work with multiple families of products
  • When you want to enforce consistency among related products
  • Cross-platform UI toolkits (Windows vs macOS widgets)
  • Database driver families (MySQL vs PostgreSQL)

💻 Java Implementation

Cross-Platform UI Factory
// ── Product Interfaces ──────────────────────────────────────
public interface Button {
    void render();
    void onClick();
}

public interface TextField {
    void render();
    String getValue();
}

public interface Checkbox {
    void render();
    boolean isChecked();
}

// ── Windows Family ──────────────────────────────────────────
public class WindowsButton implements Button {
    public void render() { System.out.println("[Windows] Rendering flat button"); }
    public void onClick() { System.out.println("[Windows] Button clicked — Win32 event"); }
}

public class WindowsTextField implements TextField {
    private String value = "";
    public void render() { System.out.println("[Windows] Rendering Win32 text field"); }
    public String getValue() { return value; }
}

public class WindowsCheckbox implements Checkbox {
    private boolean checked = false;
    public void render() { System.out.println("[Windows] Rendering Win32 checkbox"); }
    public boolean isChecked() { return checked; }
}

// ── macOS Family ────────────────────────────────────────────
public class MacButton implements Button {
    public void render() { System.out.println("[macOS] Rendering Aqua button"); }
    public void onClick() { System.out.println("[macOS] Button clicked — Cocoa event"); }
}

public class MacTextField implements TextField {
    private String value = "";
    public void render() { System.out.println("[macOS] Rendering NSTextField"); }
    public String getValue() { return value; }
}

public class MacCheckbox implements Checkbox {
    private boolean checked = false;
    public void render() { System.out.println("[macOS] Rendering NSButton (checkbox)"); }
    public boolean isChecked() { return checked; }
}

// ── Abstract Factory Interface ───────────────────────────────
public interface UIComponentFactory {
    Button createButton();
    TextField createTextField();
    Checkbox createCheckbox();
}

// ── Concrete Factories ────────────────────────────────────────
public class WindowsUIFactory implements UIComponentFactory {
    @Override public Button createButton()       { return new WindowsButton(); }
    @Override public TextField createTextField()  { return new WindowsTextField(); }
    @Override public Checkbox createCheckbox()    { return new WindowsCheckbox(); }
}

public class MacUIFactory implements UIComponentFactory {
    @Override public Button createButton()       { return new MacButton(); }
    @Override public TextField createTextField()  { return new MacTextField(); }
    @Override public Checkbox createCheckbox()    { return new MacCheckbox(); }
}

// ── Application — uses abstract factory ─────────────────────
public class LoginForm {
    private final Button submitButton;
    private final TextField usernameField;
    private final Checkbox rememberMe;

    // Injected factory — doesn't know if Windows or Mac
    public LoginForm(UIComponentFactory factory) {
        this.submitButton  = factory.createButton();
        this.usernameField = factory.createTextField();
        this.rememberMe    = factory.createCheckbox();
    }

    public void render() {
        usernameField.render();
        rememberMe.render();
        submitButton.render();
    }

    public void submit() {
        System.out.println("User: " + usernameField.getValue() + " | Remember: " + rememberMe.isChecked());
        submitButton.onClick();
    }
}

// ── Client / Configuration ────────────────────────────────────
public class Application {
    public static void main(String[] args) {
        // Determine factory based on OS at startup — only place that knows OS
        UIComponentFactory factory = getFactory();

        LoginForm form = new LoginForm(factory); // doesn't care about platform
        form.render();
        form.submit();
    }

    private static UIComponentFactory getFactory() {
        String os = System.getProperty("os.name").toLowerCase();
        if (os.contains("mac")) return new MacUIFactory();
        return new WindowsUIFactory(); // default
    }
}

📊 Pros & Cons

AspectDetail
✅ Product consistencyEnsures products of the same family work together
✅ Decouples clientClient code works with interfaces, not concrete classes
✅ Easy to switch familiesSwap entire UI family with one factory change
❌ Hard to extendAdding a new product type requires changes in ALL factories
❌ ComplexMany interfaces and classes for simple scenarios

🎤 Interview Q&A

Factory Method: One factory, one product. Uses inheritance — subclass overrides a factory method to change the product.
Abstract Factory: One factory interface, MULTIPLE related products (a family). Uses composition — clients are injected with a factory and call methods to get each product. Abstract Factory is like a "kit" of coordinated products.

To add a new platform (e.g., Linux), you create a LinuxUIFactory and Linux component classes — zero changes to existing factories, client code, or product interfaces. The client code (LoginForm) requires no changes because it depends on the abstract UIComponentFactory interface.

(1) Spring's BeanFactory / ApplicationContext — abstract factories for bean creation. (2) JDBC's Connection, Statement, ResultSet — each driver returns its own family. (3) JPA — EntityManagerFactory creates coordinated EntityManager, Query etc. (4) Spring's TransactionProxyFactoryBean family.
