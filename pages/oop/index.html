OOP Concepts in Java

Object-Oriented Programming is the foundation of Java. Master the 4 pillars — Encapsulation, Inheritance, Polymorphism, and Abstraction — with practical examples and interview answers.

Encapsulation

Definition: Encapsulation bundles data (fields) and methods that operate on the data into a single unit (class), and restricts direct access to some of the object's components using access modifiers.

When to Use

  • When you need to protect internal data from external corruption
  • When you want to control how data is accessed or modified
  • When implementing JavaBeans, DTOs, or entities
BankAccount.java 
public class BankAccount {
    // Private fields — data is hidden from outside
    private String accountNumber;
    private double balance;
    private String owner;

    // Constructor
    public BankAccount(String accountNumber, String owner, double initialBalance) {
        this.accountNumber = accountNumber;
        this.owner = owner;
        // Validation inside constructor — control over initialization
        if (initialBalance < 0) throw new IllegalArgumentException("Initial balance cannot be negative");
        this.balance = initialBalance;
    }

    // Controlled access through getter — read-only for balance
    public double getBalance() {
        return balance;
    }

    // No setter for balance — ONLY through controlled methods
    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Deposit amount must be positive");
        this.balance += amount;
        System.out.println("Deposited: " + amount + " | New Balance: " + balance);
    }

    public void withdraw(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Withdrawal must be positive");
        if (amount > balance) throw new IllegalStateException("Insufficient funds");
        this.balance -= amount;
        System.out.println("Withdrawn: " + amount + " | New Balance: " + balance);
    }

    public String getAccountNumber() { return accountNumber; }
    public String getOwner() { return owner; }

    // Main — Demo
    public static void main(String[] args) {
        BankAccount account = new BankAccount("ACC-001", "Alice", 1000.0);
        account.deposit(500);      //  Deposited: 500.0 | New Balance: 1500.0
        account.withdraw(200);     //  Withdrawn: 200.0 | New Balance: 1300.0
        // account.balance = 9999; //  Compile error — private field
    }
}

Key Points

  • Use private for fields, public for controlled getters/setters
  • Validation logic lives in setters/methods — protects invariants
  • Java's record (Java 16+) provides auto-encapsulation: record Point(int x, int y) {}
  • Real-world usage: JPA entities, Spring beans, all Java standard library classes

Interview Tip: Encapsulation ≠ just using getters/setters. The real goal is data integrity. A bank account with a public setBalance() is not properly encapsulated.

Interview Q&A

Encapsulation is the mechanism of hiding the internal state of an object and requiring all interaction to be performed through well-defined methods. We need it to: (1) protect data integrity by preventing illegal state, (2) reduce coupling between modules, (3) allow internal implementation to change without breaking external code.

Data hiding is a subset of encapsulation — it specifically refers to making fields private. Encapsulation is the broader concept: bundling data with methods AND restricting access. You can have data hiding without encapsulation (private fields with no meaningful methods), but proper encapsulation implies meaningful data hiding.

Yes. Encapsulation is about controlled access. Using private fields and only exposing behavior-based methods (like deposit(), withdraw()) achieves encapsulation without traditional getters/setters. Over-exposing via setters (like setBalance()) can actually break encapsulation.

Both promote cohesion. Encapsulation ensures a class owns its data and the logic to manipulate it. SRP ensures a class has only one reason to change. Together, they encourage classes that are tightly focused around a single concept with well-guarded internals. Breaking encapsulation (e.g., exposing all fields) often leads to violating SRP as external classes begin managing the state.

Inheritance

Definition: Inheritance lets a child class acquire properties and methods from a parent class using the extends keyword. It promotes code reuse and establishes an IS-A relationship.

Animal Hierarchy - Base (Parent) class
// Base (Parent) class
public abstract class Animal {
    protected String name;
    protected int age;

    public Animal(String name, int age) {
        this.name = name;
        this.age = age;
    }

    // Common behavior — inherited by all children
    public void breathe() {
        System.out.println(name + " is breathing.");
    }

    // Abstract method — each subclass MUST provide its own implementation
    public abstract String makeSound();

    public String getInfo() {
        return name + " (age: " + age + ")";
    }
}
Animal Hierarchy - Child class 1
// Child class 1
public class Dog extends Animal {
    private String breed;

    public Dog(String name, int age, String breed) {
        super(name, age);  // calls parent constructor
        this.breed = breed;
    }

    @Override
    public String makeSound() {
        return "Woof! Woof!";
    }

    // Dog-specific method
    public void fetch() {
        System.out.println(name + " fetches the ball!");
    }

    @Override
    public String getInfo() {
        return super.getInfo() + " | Breed: " + breed; // extend parent method
    }
}
Animal Hierarchy - Child class 2
// Child class 2
public class Cat extends Animal {
    private boolean isIndoor;

    public Cat(String name, int age, boolean isIndoor) {
        super(name, age);
        this.isIndoor = isIndoor;
    }

    @Override
    public String makeSound() {
        return "Meow!";
    }
}
Animal Hierarchy - Demo
// Demo
public class Main {
    public static void main(String[] args) {
        Dog dog = new Dog("Rex", 3, "Labrador");
        Cat cat = new Cat("Whiskers", 5, true);

        dog.breathe();             // Inherited from Animal
        System.out.println(dog.makeSound());  // Woof! Woof!
        System.out.println(dog.getInfo());    // Rex (age: 3) | Breed: Labrador
        dog.fetch();               // Dog-specific

        System.out.println(cat.makeSound());  // Meow!
    }
}

Prefer Composition over Inheritance when the relationship is HAS-A rather than IS-A. Deep inheritance hierarchies lead to fragile, hard-to-maintain code. Java only supports single inheritance for classes (but multiple interface implementation).

Interview Q&A

To avoid the Diamond Problem: if class C inherits from both A and B, and both A and B have a method foo(), the JVM wouldn't know which implementation C should use. Java solves this by allowing multiple interface inheritance (with explicit override requirements when conflicts arise) but only single class inheritance.

Overriding: Runtime polymorphism. Child class redefines a method from the parent with the same signature. Resolved at runtime. Requires @Override annotation (optional but best practice).
Overloading: Compile-time polymorphism. Same class has multiple methods with the same name but different parameter lists. Resolved at compile time by the compiler based on argument types.

No. Constructors are not inherited. However, the child class implicitly calls the parent's no-arg constructor as the first statement. To call a specific parent constructor, use super(...) as the first line of the child constructor. This is why if a parent defines only a parameterized constructor, the child must explicitly call super(args).

IS-A (Inheritance): Dog extends Animal — a Dog IS an Animal. Use when the child is truly a specialized version of the parent.
HAS-A (Composition): Car has an Engine — a Car HAS an Engine but is not an Engine. Composition is generally preferred for flexibility and avoiding tight coupling.

Polymorphism

Definition: Polymorphism allows objects of different types to be treated through a common interface. "One interface, many implementations." Two types: Compile-time (overloading) and Runtime (overriding).

Runtime Polymorphism - Polymorphism with a Payment system
// Polymorphism with a Payment system
public interface PaymentProcessor {
    void processPayment(double amount);
    default String getCurrency() { return "USD"; }
}

public class CreditCardProcessor implements PaymentProcessor {
    @Override
    public void processPayment(double amount) {
        System.out.println("Processing credit card payment: $" + amount);
        // charge creditcard gateway
    }
}

public class PayPalProcessor implements PaymentProcessor {
    @Override
    public void processPayment(double amount) {
        System.out.println("Processing PayPal payment: $" + amount);
        // call PayPal API
    }
}

public class UPIProcessor implements PaymentProcessor {
    @Override
    public void processPayment(double amount) {
        System.out.println("Processing UPI payment: ₹" + amount);
    }
    @Override
    public String getCurrency() { return "INR"; }
}
Runtime Polymorphism - Client code — works with ANY PaymentProcessor
// Client code — works with ANY PaymentProcessor
public class Checkout {
    // Runtime polymorphism: the actual type is determined at runtime
    public void checkout(PaymentProcessor processor, double amount) {
        System.out.println("Currency: " + processor.getCurrency());
        processor.processPayment(amount);  // which processPayment? Decided at RUNTIME
    }

    public static void main(String[] args) {
        Checkout checkout = new Checkout();

        PaymentProcessor card = new CreditCardProcessor();
        PaymentProcessor paypal = new PayPalProcessor();
        PaymentProcessor upi = new UPIProcessor();

        checkout.checkout(card, 99.99);     // Processing credit card payment: $99.99
        checkout.checkout(paypal, 49.99);   // Processing PayPal payment: $49.99
        checkout.checkout(upi, 500.0);      // Processing UPI payment: ₹500.0

        // Compile-time polymorphism (overloading)
        Calculator calc = new Calculator();
        System.out.println(calc.add(1, 2));         // int version
        System.out.println(calc.add(1.5, 2.5));     // double version
        System.out.println(calc.add(1, 2, 3));      // three args version
    }
}
Runtime Polymorphism - Compile-time polymorphism — method overloading
// Compile-time polymorphism — method overloading
class Calculator {
    public int add(int a, int b) { return a + b; }
    public double add(double a, double b) { return a + b; }
    public int add(int a, int b, int c) { return a + b + c; }
}

Interview Q&A

Compile-time (Static Binding): Resolved by the compiler. Achieved through method overloading. The method to call is determined at compile time based on the argument types.
Runtime (Dynamic Binding): Resolved by the JVM at runtime. Achieved through method overriding. The JVM looks at the actual object type (not the reference type) to determine which method to invoke. This is what makes polymorphism powerful — you can write code against an interface and swap implementations.

No. Static methods are resolved at compile time (static binding) based on the reference type, not the object type. This is called method hiding, not overriding. If a child class defines the same static method, it hides the parent's method. @Override annotation will cause a compile error on static methods.

Yes — through interfaces. Two unrelated classes can implement the same interface and be used polymorphically. This is actually preferred in Java: "program to an interface, not an implementation." Java also supports duck typing via reflection and lambda expressions for functional interfaces.

Abstraction

Definition: Abstraction hides implementation details and shows only the essential features. Achieved via abstract classes and interfaces.

FeatureAbstract ClassInterface
InstantiationCannot be instantiatedCannot be instantiated
MethodsCan have abstract + concreteabstract, default, static (Java 8+)
FieldsAny type of fieldOnly public static final constants
InheritanceSingle class inheritanceMultiple interface implementation
ConstructorCan have constructorsNo constructors
Use whenSharing code among related classesDefining a contract for unrelated classes
Vehicle Abstraction - Abstract class — partial abstraction (some concrete methods)
// Abstract class — partial abstraction (some concrete methods)
public abstract class Vehicle {
    protected String brand;
    protected int year;

    public Vehicle(String brand, int year) {
        this.brand = brand;
        this.year = year;
    }

    // Concrete method — shared by all vehicles
    public void startEngine() {
        System.out.println(brand + " engine started.");
    }

    // Abstract methods — each vehicle defines its own behavior
    public abstract double calculateFuelCost(double km);
    public abstract String getVehicleType();
}
Vehicle Abstraction - Interface — full abstraction / contract
// Interface — full abstraction / contract
public interface GPS {
    void getCurrentLocation();
    default void navigate(String destination) {
        System.out.println("Navigating to: " + destination);
    }
}

public interface ElectricVehicle {
    void chargeBattery();
    int getBatteryLevel();
}
Vehicle Abstraction - Concrete class implementing everything
// Concrete class implementing everything
public class TeslaModel3 extends Vehicle implements GPS, ElectricVehicle {
    private int batteryLevel;

    public TeslaModel3(int year, int batteryLevel) {
        super("Tesla", year);
        this.batteryLevel = batteryLevel;
    }

    @Override
    public double calculateFuelCost(double km) {
        // 0.15 kWh per km, $0.12 per kWh
        return km * 0.15 * 0.12;
    }

    @Override
    public String getVehicleType() { return "Electric Sedan"; }

    @Override
    public void getCurrentLocation() {
        System.out.println("Getting GPS location via satellite...");
    }

    @Override
    public void chargeBattery() {
        batteryLevel = 100;
        System.out.println("Battery fully charged!");
    }

    @Override
    public int getBatteryLevel() { return batteryLevel; }

    public static void main(String[] args) {
        TeslaModel3 tesla = new TeslaModel3(2024, 80);
        tesla.startEngine();              // Inherited from Vehicle
        tesla.getCurrentLocation();       // From GPS
        tesla.navigate("Times Square");   // Default method from GPS
        System.out.printf("Cost for 100km: $%.2f%n", tesla.calculateFuelCost(100));
    }
}

Interview Q&A

Use an abstract class when: (1) related classes share common code, (2) you need constructor logic, (3) fields are not constants, (4) you want to provide a default implementation that subclasses can override.
Use an interface when: (1) unrelated classes need the same contract (e.g., Comparable, Serializable), (2) you need multiple type inheritance, (3) you want to define a pure API without implementation details.

Default methods allow interfaces to have concrete method implementations using the default keyword. They were introduced in Java 8 to allow backward-compatible evolution of interfaces (e.g., List.sort() was added as a default method without breaking all existing List implementors). If a class implements two interfaces with the same default method, it must override it to resolve the conflict.

No, because Java has primitive types (int, long, double, etc.) which are not objects. Wrapper classes (Integer, Long) provide object representations via autoboxing/unboxing. Additionally, static methods and fields don't belong to an object instance. Languages like Smalltalk and Ruby are considered more "purely" object-oriented.
