SOLID Principles in Java

SOLID is a set of 5 design principles for writing maintainable, scalable, and testable object-oriented code. Coined by Robert C. Martin (Uncle Bob).

S — Single Responsibility Principle

"A class should have one, and only one, reason to change." — A class should do one thing and do it well. If a class handles multiple concerns, changes to one concern may break the other.

❌ Violating SRP
// BAD: This class has 3 responsibilities
public class UserService {
    public void createUser(User user) { /* DB logic */ }
    public String generateReport(User user) { /* Reporting logic */ }
    public void sendWelcomeEmail(User user) { /* Email logic */ }
    // If email logic changes, this class changes. If DB logic changes, this class changes too.
}
✅ Following SRP
// Each class has ONE reason to change

// Responsibility 1: User persistence
public class UserRepository {
    public void save(User user) {
        // Only database interaction logic here
        System.out.println("Saving user to DB: " + user.getName());
    }
}

// Responsibility 2: Report generation
public class UserReportService {
    public String generateReport(User user) {
        // Only reporting logic here
        return "User Report for: " + user.getName();
    }
}

// Responsibility 3: Email sending
public class EmailService {
    public void sendWelcomeEmail(User user) {
        // Only email logic here
        System.out.println("Sending welcome email to: " + user.getEmail());
    }
}

// Orchestration: UserService now just coordinates
public class UserService {
    private final UserRepository repo;
    private final EmailService emailService;

    public UserService(UserRepository repo, EmailService emailService) {
        this.repo = repo;
        this.emailService = emailService;
    }

    public void registerUser(User user) {
        repo.save(user);
        emailService.sendWelcomeEmail(user);
        // Each collaborator can change independently
    }
}

Look for: (1) classes with more than one "reason to change" — ask "why would this class change?", (2) large classes with many methods doing unrelated things, (3) method names that don't share a common theme (e.g., save(),generatePDF(),sendEmail() in the same class), (4) classes that need to import many unrelated packages.

Yes — over-application can cause "class explosion" and indirection. The key is balance. A class should be small and focused, but splitting trivial togethers creates unnecessary complexity. SRP is about cohesion: methods in a class should all work towards the same purpose.

O — Open/Closed Principle

"Software entities should be open for extension, but closed for modification." — Add new functionality by creating new code, not changing existing tested code.

✅ OCP with Strategy Pattern
// Closed for modification — shape interface never changes
public interface Shape {
    double area();
}

// Open for extension — add new shapes without touching existing code
public class Circle implements Shape {
    private double radius;
    public Circle(double radius) { this.radius = radius; }
    @Override
    public double area() { return Math.PI * radius * radius; }
}

public class Rectangle implements Shape {
    private double width, height;
    public Rectangle(double w, double h) { this.width = w; this.height = h; }
    @Override
    public double area() { return width * height; }
}

// NEW shape added — zero changes to existing code!
public class Triangle implements Shape {
    private double base, height;
    public Triangle(double base, double height) { this.base = base; this.height = height; }
    @Override
    public double area() { return 0.5 * base * height; }
}

// AreaCalculator is CLOSED for modification
public class AreaCalculator {
    public double totalArea(List<Shape> shapes) {
        return shapes.stream().mapToDouble(Shape::area).sum();
    }
}

// Usage
List<Shape> shapes = List.of(new Circle(5), new Rectangle(4, 6), new Triangle(3, 8));
System.out.println(new AreaCalculator().totalArea(shapes)); // Works for ANY shape!

Spring uses OCP extensively: (1) Dependency Injection — inject different implementations without changing consumers, (2) AOP (Aspect-Oriented Programming) — add cross-cutting concerns (logging, security) without modifying business classes, (3) Strategy Pattern in AuthenticationProvider, HandlerMapping etc. — swap implementations via configuration.

No — no class can be closed to all possible changes. The goal is to anticipate the most likely axes of change and use abstractions to protect against them. You can't predict every future requirement, but OCP encourages identifying variation points and encapsulating them behind interfaces or abstract classes.

L — Liskov Substitution Principle

"Subtypes must be substitutable for their base types." — If S extends T, then objects of type T may be replaced with objects of type S without altering correctness.

❌ LSP Violation — Square extends Rectangle
// Classic LSP violation
class Rectangle {
    protected int width, height;
    public void setWidth(int w) { this.width = w; }
    public void setHeight(int h) { this.height = h; }
    public int area() { return width * height; }
}

// PROBLEM: Square overrides setters to maintain equality — violates Rectangle's contract
class Square extends Rectangle {
    @Override
    public void setWidth(int w) { this.width = w; this.height = w; } // ← forces height = width
    @Override
    public void setHeight(int h) { this.height = h; this.width = h; } // ← forces width = height
}

// This code BREAKS when a Square is passed where Rectangle is expected
void testRectangle(Rectangle r) {
    r.setWidth(5);
    r.setHeight(4);
    assert r.area() == 20; // ✅ for Rectangle, ❌ fails for Square (returns 16)
}
✅ LSP Fix — Common base Shape
// Fix: Don't inherit if it violates substitutability
public interface Shape {
    int area();
}

public class Rectangle implements Shape {
    private int width, height;
    public Rectangle(int w, int h) { this.width = w; this.height = h; }
    public int area() { return width * height; }
}

public class Square implements Shape {
    private int side;
    public Square(int side) { this.side = side; }
    public int area() { return side * side; }
}
// Both are Shapes — each is substitutable for Shape — no violation!

(1) If a subclass throws UnsupportedOperationException for methods inherited from the parent — it's an LSP violation (common with the Collections.unmodifiableList()-style pattern). (2) If client code checks instanceof before calling methods — violation likely. (3) If a subclass weakens preconditions or strengthens postconditions of the parent contract. (4) If unit tests written for the parent class fail when a subclass is substituted.

OCP says "extend without modifying." LSP ensures that extensions (subclasses) are truly compatible replacements. Without LSP, OCP breaks down — extending a class creates a subclass that can't be used as a drop-in replacement, forcing changes in client code. They work together: OCP defines the goal, LSP defines the correctness constraint on extensions.

I — Interface Segregation Principle

"Clients should not be forced to depend on methods they do not use." — Split large interfaces into smaller, more specific ones.

✅ ISP — Segregated Interfaces
// BAD: Fat interface forces all implementors to implement ALL methods
interface Worker {
    void work();
    void eat();    // ← Robots don't eat! Forced to implement with throws
    void sleep();  // ← Robots don't sleep!
}

// GOOD: Segregated interfaces
interface Workable {
    void work();
}

interface Eatable {
    void eat();
}

interface Sleepable {
    void sleep();
}

// HumanWorker needs all three
public class HumanWorker implements Workable, Eatable, Sleepable {
    public void work()  { System.out.println("Human working"); }
    public void eat()   { System.out.println("Human eating"); }
    public void sleep() { System.out.println("Human sleeping"); }
}

// RobotWorker only needs Workable — clean!
public class RobotWorker implements Workable {
    public void work() { System.out.println("Robot working 24/7"); }
    // No forced eat() or sleep() implementations!
}

// Real-world: Spring Data JPA interfaces are segregated
// CrudRepository, PagingAndSortingRepository, JpaRepository — each adds more contract

Examples: (1) Readable, Writable, Closeable — each is a separate interface rather than one Stream interface forcing all. (2) Iterable vs Collection vs List — progressively more capable. (3) Runnable vs Callable — separate for different async needs. (4) Spring's ApplicationEventPublisher, ApplicationContext — different capability layers.

D — Dependency Inversion Principle

"High-level modules should not depend on low-level modules. Both should depend on abstractions." — Depend on interfaces/abstract classes, not concrete implementations.

✅ DIP with Spring Dependency Injection
// Abstraction (interface) — both high and low-level depend on this
public interface NotificationService {
    void send(String message, String recipient);
}

// Low-level module 1
public class EmailNotificationService implements NotificationService {
    @Override
    public void send(String message, String recipient) {
        System.out.println("EMAIL to " + recipient + ": " + message);
    }
}

// Low-level module 2
public class SMSNotificationService implements NotificationService {
    @Override
    public void send(String message, String recipient) {
        System.out.println("SMS to " + recipient + ": " + message);
    }
}

// High-level module — depends on ABSTRACTION, not concrete class
@Service  // Spring manages this
public class OrderService {
    // Inject via interface — Spring will provide the actual implementation
    private final NotificationService notificationService;

    // Constructor injection (preferred)
    @Autowired
    public OrderService(NotificationService notificationService) {
        this.notificationService = notificationService;
    }

    public void placeOrder(Order order) {
        // Process order...
        // High-level module doesn't know or care if it's Email or SMS
        notificationService.send("Order confirmed: #" + order.getId(), order.getCustomerEmail());
    }
}

// Spring configuration — swap implementations via config, no code changes!
@Configuration
public class AppConfig {
    @Bean
    public NotificationService notificationService() {
        // Switch between Email, SMS, Push — high-level code unchanged
        return new EmailNotificationService();
    }
}

Dependency Inversion Principle (DIP) is the concept: high-level modules should depend on abstractions. Dependency Injection (DI) is the technique to achieve it: provide dependencies from outside rather than creating them internally (constructor injection, setter injection, field injection). IoC containers like Spring implement DI to facilitate DIP.

(1) Immutability: Constructor-injected fields can be final. (2) Testability: Inject mocks directly in unit tests without Spring context. (3) Detecting circular dependencies at startup. (4) Explicit contracts: All required dependencies are visible in the constructor signature. Field injection hides dependencies and requires reflection.
