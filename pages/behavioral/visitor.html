Visitor Pattern

Lets you add further operations to objects without modifying them. Separates algorithms from the objects on which they operate. Uses double dispatch.

When to Use

  • When you need to add many distinct operations to an object structure
  • When the object structure rarely changes, but you frequently add new operations
  • AST traversal in compilers, document rendering, tax calculations across different item types

Java Implementation

Tax Calculator Visitor - Visitor interface — one method per element type
// Visitor interface — one method per element type
public interface TaxVisitor {
    double visit(FoodItem item);
    double visit(ElectronicsItem item);
    double visit(LuxuryItem item);
}
Tax Calculator Visitor - Element interface — accepts any visitor
// Element interface — accepts any visitor
public interface ShoppingItem {
    String getName();
    double getPrice();
    double accept(TaxVisitor visitor); // double dispatch
}
Tax Calculator Visitor - Concrete elements
// Concrete elements
public class FoodItem implements ShoppingItem {
    private final String name;
    private final double price;
    public FoodItem(String name, double price) { this.name = name; this.price = price; }
    public String getName()  { return name; }
    public double getPrice() { return price; }

    @Override
    public double accept(TaxVisitor visitor) {
        return visitor.visit(this); // dispatch to correct visitor method
    }
}

public class ElectronicsItem implements ShoppingItem {

Tax Calculator Visitor - Concrete Visitor 1 — Standard Tax
// Concrete Visitor 1 — Standard Tax
public class StandardTaxVisitor implements TaxVisitor {
    public double visit(FoodItem item)       { return item.getPrice() * 0.05; }  // 5% GST
    public double visit(ElectronicsItem i)   { return i.getPrice() * 0.18; }     // 18% GST
    public double visit(LuxuryItem item)     { return item.getPrice() * 0.28; }  // 28% GST
}
Tax Calculator Visitor - Concrete Visitor 2 — Export Tax (add operations WITHOUT modifying elements)
// Concrete Visitor 2 — Export Tax (add operations WITHOUT modifying elements)
public class ExportTaxVisitor implements TaxVisitor {
    public double visit(FoodItem item)       { return item.getPrice() * 0.10; }
    public double visit(ElectronicsItem i)   { return i.getPrice() * 0.25; }
    public double visit(LuxuryItem item)     { return item.getPrice() * 0.40; }
}
Tax Calculator Visitor - Usage
// Usage
import java.util.List;
public class Main {
    public static void main(String[] args) {
        List<ShoppingItem> cart = List.of(
            new FoodItem("Rice (5kg)", 200),
            new ElectronicsItem("Laptop", 50000),
            new LuxuryItem("Designer Watch", 150000)
        );

        TaxVisitor standardTax = new StandardTaxVisitor();
        TaxVisitor exportTax   = new ExportTaxVisitor();

        System.out.println("=== Standard Tax ===");
        cart.forEach(item -> {
            double tax = item.accept(standardTax); // polymorphic dispatch
            System.out.printf("%s: Price=%.0f Tax=%.0f Total=%.0f%n",
                item.getName(), item.getPrice(), tax, item.getPrice() + tax);
        });

        System.out.println("\n=== Export Tax ===");
        double total = cart.stream().mapToDouble(item -> {
            double tax = item.accept(exportTax);
            System.out.printf("%s: Tax=%.0f%n", item.getName(), tax);
            return item.getPrice() + tax;
        }).sum();
        System.out.println("Total: " + total);
    }
}

Interview Q&A

In single dispatch (standard Java polymorphism), the method called depends on the runtime type of the object. In double dispatch, the method called depends on the runtime types of both the object AND the argument. Visitor achieves this: item.accept(visitor) dispatches on item type; inside accept(), visitor.visit(this) dispatches on visitor type — two runtime types determine the method.

Adding a new element type requires updating ALL visitors — violates OCP from the element's perspective. Visitor is a trade-off: it's OCP-friendly for adding operations, but OCP-unfriendly for adding new element types. Use it only when the element hierarchy is stable and new operations are frequent. Java 17+ sealed classes + pattern matching (switch) can replace Visitor in some cases.
