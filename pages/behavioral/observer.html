19. Observer Pattern

Defines a one-to-many dependency between objects so that when one object (Subject) changes state, all its dependents (Observers) are notified and updated automatically.

When to Use

  • Event handling systems (GUI events, DOM events)
  • Pub/Sub messaging systems (Kafka, RabbitMQ)
  • Model-View-Controller (MVC) — model notifies views
  • Spring's ApplicationEvent system
  • Stock price notification systems

Java Implementation

E-Commerce Order Event System
import java.util.ArrayList;
import java.util.List;
E-Commerce Order Event System - Event Object
//  Event Object 
public class OrderEvent {
    public enum Type { PLACED, SHIPPED, DELIVERED, CANCELLED }
    private final String orderId;
    private final Type type;
    private final double amount;

    public OrderEvent(String orderId, Type type, double amount) {
        this.orderId = orderId;
        this.type    = type;
        this.amount  = amount;
    }
    public String getOrderId() { return orderId; }
    public Type getType()      { return type; }
    public double getAmount()  { return amount; }
}
E-Commerce Order Event System - Observer Interface
//  Observer Interface 
public interface OrderObserver {
    void onOrderEvent(OrderEvent event);
}
E-Commerce Order Event System - Subject Interface
//  Subject Interface 
public interface OrderSubject {
    void subscribe(OrderObserver observer);
    void unsubscribe(OrderObserver observer);
    void notifyObservers(OrderEvent event);
}
E-Commerce Order Event System - Concrete Subject
//  Concrete Subject 
public class OrderService implements OrderSubject {
    private final List<OrderObserver> observers = new ArrayList<>();

    @Override
    public void subscribe(OrderObserver observer) {
        observers.add(observer);
        System.out.println(" Observer registered: " + observer.getClass().getSimpleName());
    }

    @Override
    public void unsubscribe(OrderObserver observer) {
        observers.remove(observer);
    }

    @Override
    public void notifyObservers(OrderEvent event) {
        observers.forEach(obs -> obs.onOrderEvent(event));
    }

    // Business method — triggers notifications
    public void placeOrder(String orderId, double amount) {
        System.out.println("\n Order placed: " + orderId);
        notifyObservers(new OrderEvent(orderId, OrderEvent.Type.PLACED, amount));
    }

    public void shipOrder(String orderId) {
        System.out.println("\n Order shipped: " + orderId);
        notifyObservers(new OrderEvent(orderId, OrderEvent.Type.SHIPPED, 0));
    }
}
E-Commerce Order Event System - Concrete Observers
//  Concrete Observers 
public class EmailNotificationObserver implements OrderObserver {
    @Override
    public void onOrderEvent(OrderEvent event) {
        System.out.println(" EMAIL: Order " + event.getOrderId()
            + " is " + event.getType() + " (Amount: $" + event.getAmount() + ")");
    }
}

public class SMSNotificationObserver implements OrderObserver {
    @Override
    public void onOrderEvent(OrderEvent event) {
        if (event.getType() == OrderEvent.Type.SHIPPED) { // only care about shipping
            System.out.println(" SMS: Your order " + event.getOrderId() + " has been shipped!");
        }
    }
}

public class InventoryObserver implements OrderObserver {
    @Override
    public void onOrderEvent(OrderEvent event) {
        if (event.getType() == OrderEvent.Type.PLACED) {
            System.out.println(" INVENTORY: Reducing stock for order " + event.getOrderId());
        }
    }
}

public class AnalyticsObserver implements OrderObserver {
    private int totalOrders = 0;
    private double totalRevenue = 0;

    @Override
    public void onOrderEvent(OrderEvent event) {
        if (event.getType() == OrderEvent.Type.PLACED) {
            totalOrders++;
            totalRevenue += event.getAmount();
            System.out.println(" ANALYTICS: Orders=" + totalOrders + " Revenue=$" + totalRevenue);
        }
    }
}
E-Commerce Order Event System - Usage
// Usage
public class Main {
    public static void main(String[] args) {
        OrderService orderService = new OrderService();

        // Register observers — loose coupling, easy to add/remove
        orderService.subscribe(new EmailNotificationObserver());
        orderService.subscribe(new SMSNotificationObserver());
        orderService.subscribe(new InventoryObserver());
        orderService.subscribe(new AnalyticsObserver());

        // Subject changes state — all observers notified
        orderService.placeOrder("ORD-001", 99.99);
        orderService.shipOrder("ORD-001");
        orderService.placeOrder("ORD-002", 149.50);
    }
}

Spring Framework — Built-in Observer via ApplicationEvent

Spring Events (Production-Grade) - Event
// Event
public class OrderPlacedEvent extends ApplicationEvent {
    private final String orderId;
    public OrderPlacedEvent(Object source, String orderId) {
        super(source);
        this.orderId = orderId;
    }
    public String getOrderId() { return orderId; }
}
Spring Events (Production-Grade) - Publisher (Subject)
// Publisher (Subject)
@Service
public class OrderService {
    @Autowired private ApplicationEventPublisher publisher;

    public void placeOrder(String orderId) {
        // business logic...
        publisher.publishEvent(new OrderPlacedEvent(this, orderId));
    }
}
Spring Events (Production-Grade) - Listener (Observer)
// Listener (Observer)
@Component
public class EmailListener {
    @EventListener
    public void handleOrderPlaced(OrderPlacedEvent event) {
        System.out.println("Sending email for order: " + event.getOrderId());
    }
}
Spring Events (Production-Grade) - Async Listener
// Async Listener
@Component
public class AnalyticsListener {
    @Async
    @EventListener
    public void handleOrderPlaced(OrderPlacedEvent event) {
        // Runs in a separate thread — non-blocking
        System.out.println("Recording analytics for: " + event.getOrderId());
    }
}

Interview Q&A

Observer: Subject and Observers know about each other (direct reference). Tightly coupled to the subject. Synchronous — observer is called directly.
Pub/Sub: Publisher and Subscribers are decoupled via a message broker (Kafka, RabbitMQ, Spring Events). Neither knows about the other. Can be asynchronous. More scalable but more infrastructure.

If observers are never unregistered (unsubscribe()), the subject holds strong references to them, preventing garbage collection — classic memory leak. Solutions: (1) Use WeakReference<Observer> in the observers list. (2) Always unsubscribe when an observer is no longer needed (e.g., component destroyed). (3) Spring manages this for you with @EventListener beans.

Reactive Programming (RxJava, Project Reactor, Spring WebFlux) is an evolved form of Observer. It provides: (1) Backpressure — observer can signal it can't keep up, (2) Composable streams with operators (map, filter, flatMap), (3) Error handling in the reactive chain, (4) Completion signals. Java's java.util.concurrent.Flow (Java 9+) standardizes this as Reactive Streams.
