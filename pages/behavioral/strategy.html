Strategy Pattern

Defines a family of algorithms, encapsulates each one, and makes them interchangeable. Strategy lets the algorithm vary independently from the clients that use it.

When to Use

  • When you have multiple variants of an algorithm
  • When you want to switch algorithms at runtime
  • When you have lots of conditional statements selecting behavior
  • Sorting strategies, payment methods, compression algorithms, discount policies

Java Implementation

E-Commerce Discount Strategy - Strategy interface — the contract for all algorithms
// Strategy interface — the contract for all algorithms
@FunctionalInterface
public interface DiscountStrategy {
    double applyDiscount(double originalPrice);

    // Default factory methods for named strategies
    static DiscountStrategy noDiscount() {
        return price -> price;
    }

    static DiscountStrategy percentage(double percent) {
        return price -> price * (1 - percent / 100);
    }

    static DiscountStrategy fixed(double amount) {
        return price -> Math.max(0, price - amount);
    }

    static DiscountStrategy buyOneGetOne() {
        return price -> price / 2; // halves the price (2 items for price of 1)
    }
}
E-Commerce Discount Strategy - Context — uses a strategy
// Context — uses a strategy
public class ShoppingCart {
    private List<Double> items = new ArrayList<>();
    private DiscountStrategy discountStrategy;

    public ShoppingCart() {
        this.discountStrategy = DiscountStrategy.noDiscount(); // default
    }

    public void setDiscountStrategy(DiscountStrategy strategy) {
        this.discountStrategy = strategy;
    }

    public void addItem(double price) { items.add(price); }

    public double totalPrice() {
        double subtotal = items.stream().mapToDouble(Double::doubleValue).sum();
        double discounted = discountStrategy.applyDiscount(subtotal);
        System.out.printf("Subtotal: $%.2f | After discount: $%.2f%n", subtotal, discounted);
        return discounted;
    }
}
E-Commerce Discount Strategy - Classic OO approach (without lambdas)
// Classic OO approach (without lambdas)
public class SeasonalDiscountStrategy implements DiscountStrategy {
    private final String season;
    public SeasonalDiscountStrategy(String season) { this.season = season; }

    @Override
    public double applyDiscount(double price) {
        return switch (season.toUpperCase()) {
            case "CHRISTMAS"  -> price * 0.70; // 30% off
            case "BLACKFRIDAY"-> price * 0.50; // 50% off
            case "NEWYEAR"    -> price * 0.85; // 15% off
            default           -> price;
        };
    }
}
E-Commerce Discount Strategy - Usage — Strategy changes at runtime
// Usage — Strategy changes at runtime
public class Main {
    public static void main(String[] args) {
        ShoppingCart cart = new ShoppingCart();
        cart.addItem(50.00);
        cart.addItem(30.00);
        cart.addItem(20.00);  // Subtotal: $100

        // No discount
        cart.setDiscountStrategy(DiscountStrategy.noDiscount());
        cart.totalPrice(); // $100.00

        // 20% off
        cart.setDiscountStrategy(DiscountStrategy.percentage(20));
        cart.totalPrice(); // $80.00

        // Fixed $15 off
        cart.setDiscountStrategy(DiscountStrategy.fixed(15));
        cart.totalPrice(); // $85.00

        // Black Friday special
        cart.setDiscountStrategy(new SeasonalDiscountStrategy("BLACKFRIDAY"));
        cart.totalPrice(); // $50.00

        // Lambda strategy — inline, no class needed
        cart.setDiscountStrategy(price -> price > 80 ? price * 0.90 : price);
        cart.totalPrice(); // $90.00 (10% off since > $80)
    }
}

Real-World Usage

  • java.util.Comparator — Strategy for sorting (passed to List.sort())
  • java.util.concurrent.ThreadPoolExecutor — rejection policies are strategies
  • Spring Security's AuthenticationStrategy
  • Spring's ResourceLoader strategies
  • Jackson's SerializationFeature settings

Interview Q&A

Strategy: The algorithm is chosen by the client and injected. The strategy doesn't know about the context. Typically set once and can be changed from outside. Intent: vary the algorithm.
State: The object changes its own strategy-like behavior based on internal state. Transitions are managed internally by the states themselves. Intent: vary behavior based on state. State pattern involves states knowing about other states; Strategy doesn't.

Comparator is a textbook Strategy pattern. List.sort(Comparator) accepts any sorting strategy. The list (Context) doesn't know how elements are compared — it delegates to the strategy. With Java 8 lambdas, strategies become even cleaner: list.sort((a, b) -> a.getName().compareTo(b.getName())). This is Strategy without needing a named class.

When Strategy is a @FunctionalInterface, Java 8+ lambdas eliminate the need for concrete strategy classes. Benefits: (1) Less boilerplate — no class file per strategy, (2) Inline definition of simple strategies, (3) Composable via andThen(), compose() on function interfaces, (4) Method references for reusing existing methods as strategies: String::compareToIgnoreCase.
