📡 Mediator Pattern

Defines an object (mediator) that encapsulates how a set of objects interact. Promotes loose coupling by keeping objects from referring to each other explicitly.

🎯 When to Use

  • When many objects communicate in complex ways (spaghetti references)
  • Chat rooms, air traffic control, event bus systems
  • Spring's ApplicationContext as mediator between beans
  • MVC frameworks — controller mediates between model and view

💻 Java Implementation

Chat Room Mediator
import java.util.ArrayList;
import java.util.List;

// Mediator interface
public interface ChatMediator {
    void sendMessage(String message, User sender);
    void addUser(User user);
}

// Colleague
public abstract class User {
    protected final ChatMediator mediator;
    protected final String name;

    public User(ChatMediator mediator, String name) {
        this.mediator = mediator;
        this.name     = name;
    }

    public abstract void receive(String message, String from);
    public void send(String message) {
        System.out.println("[" + name + " → ALL]: " + message);
        mediator.sendMessage(message, this); // via mediator, not direct reference
    }

    public String getName() { return name; }
}

// Concrete Mediator
public class ChatRoom implements ChatMediator {
    private final List<User> users = new ArrayList<>();

    @Override
    public void addUser(User user) { users.add(user); }

    @Override
    public void sendMessage(String message, User sender) {
        // Mediator controls who gets the message
        users.stream()
            .filter(u -> u != sender) // don't echo to sender
            .forEach(u -> u.receive(message, sender.getName()));
    }
}

// Concrete User
public class ChatUser extends User {
    public ChatUser(ChatMediator mediator, String name) {
        super(mediator, name);
        mediator.addUser(this);
    }

    @Override
    public void receive(String message, String from) {
        System.out.println("  [" + name + " received from " + from + "]: " + message);
    }
}

// Specialized user — only receives messages
public class AdminUser extends User {
    public AdminUser(ChatMediator mediator) {
        super(mediator, "ADMIN");
        mediator.addUser(this);
    }

    @Override
    public void receive(String message, String from) {
        System.out.println("  📋 [ADMIN monitoring from " + from + "]: " + message);
    }
}

// Usage
public class Main {
    public static void main(String[] args) {
        ChatMediator chatRoom = new ChatRoom();

        User alice = new ChatUser(chatRoom, "Alice");
        User bob   = new ChatUser(chatRoom, "Bob");
        User carol = new ChatUser(chatRoom, "Carol");
        User admin = new AdminUser(chatRoom);  // monitors all

        alice.send("Hello everyone!");
        // Bob, Carol, Admin receive — Alice doesn't
        // No direct Alice → Bob reference!

        bob.send("Hi Alice! Hi Carol!");
        // Alice, Carol, Admin receive
    }
}

🎤 Interview Q&A

Observer: One subject notifies many observers — one-to-many. Subject knows about observers.
Mediator: Many objects communicate through a central mediator — many-to-many. Objects only know the mediator, not each other. Mediator encapsulates the communication logic (routing, filtering). Observer is simpler; Mediator handles more complex inter-object communication.

Yes — the mediator can become a "God Object" that knows too much and violates SRP. Solution: keep the mediator thin — only routing logic, no business logic. For heavy load, use async messaging (Kafka, RabbitMQ) instead of a synchronous mediator.
