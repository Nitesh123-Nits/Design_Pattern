20. State Pattern

Allows an object to alter its behavior when its internal state changes. The object will appear to change its class. Eliminates large conditional statements.

When to Use

  • When an object's behavior depends on its state and must change at runtime
  • When you have many conditional branches depending on the object's state
  • Order management systems, vending machines, traffic lights, ATMs
  • Spring Statemachine framework

Java Implementation

Vending Machine - State interface
// State interface
public interface VendingMachineState {
    void insertCoin(VendingMachine machine);
    void pressButton(VendingMachine machine);
    void dispense(VendingMachine machine);
    String getStateName();
}
Vending Machine - Context
// Context
public class VendingMachine {
    private VendingMachineState state;
    private int inventory = 0;

    public VendingMachine(int inventory) {
        this.inventory = inventory;
        this.state = (inventory > 0) ? new IdleState() : new OutOfStockState();
        System.out.println("Vending Machine initialized. Inventory: " + inventory);
    }

    public void insertCoin()  { state.insertCoin(this); }
    public void pressButton() { state.pressButton(this); }
    public void dispense()    { state.dispense(this); }

    public void setState(VendingMachineState state) {
        System.out.println("State transition: " + this.state.getStateName()
            + " → " + state.getStateName());
        this.state = state;
    }

    public int getInventory() { return inventory; }
    public void releaseProduct() {
        if (inventory > 0) inventory--;
    }
}
Vending Machine - Concrete States
// Concrete States
public class IdleState implements VendingMachineState {
    @Override
    public void insertCoin(VendingMachine machine) {
        System.out.println(" Coin inserted.");
        machine.setState(new HasCoinState());
    }
    @Override public void pressButton(VendingMachine machine) { System.out.println(" Insert coin first."); }
    @Override public void dispense(VendingMachine machine)    { System.out.println(" Insert coin first."); }
    @Override public String getStateName() { return "IDLE"; }
}

public class HasCoinState implements VendingMachineState {
    @Override public void insertCoin(VendingMachine machine) { System.out.println(" Coin already inserted."); }
    @Override
    public void pressButton(VendingMachine machine) {
        System.out.println(" Button pressed.");
        machine.setState(new DispensingState());
    }
    @Override public void dispense(VendingMachine machine) { System.out.println(" Press button first."); }
    @Override public String getStateName() { return "HAS_COIN"; }
}

public class DispensingState implements VendingMachineState {
    @Override public void insertCoin(VendingMachine machine) { System.out.println(" Please wait, dispensing..."); }
    @Override public void pressButton(VendingMachine machine) { System.out.println(" Already dispensing..."); }
    @Override
    public void dispense(VendingMachine machine) {
        machine.releaseProduct();
        System.out.println(" Product dispensed.");
        if (machine.getInventory() > 0) {
            machine.setState(new IdleState());
        } else {
            machine.setState(new OutOfStockState());
        }
    }
    @Override public String getStateName() { return "DISPENSING"; }
}

public class OutOfStockState implements VendingMachineState {
    @Override public void insertCoin(VendingMachine machine) { System.out.println(" Machine out of stock."); }
    @Override public void pressButton(VendingMachine machine) { System.out.println(" Machine out of stock."); }
    @Override public void dispense(VendingMachine machine)    { System.out.println(" Machine out of stock."); }
    @Override public String getStateName() { return "OUT_OF_STOCK"; }
}
Vending Machine - Usage
// Usage
public class Main {
    public static void main(String[] args) {
        VendingMachine vm = new VendingMachine(2); // Start with 2 items

        vm.insertCoin();   // IDLE → HAS_COIN
        vm.pressButton();  // HAS_COIN → DISPENSING
        vm.dispense();     // DISPENSING → IDLE (1 item left)

        vm.pressButton();  // Insert coin first
        vm.insertCoin();   // IDLE → HAS_COIN
        vm.pressButton();  // HAS_COIN → DISPENSING
        vm.dispense();     // DISPENSING → OUT_OF_STOCK (0 items left)

        vm.insertCoin();   // Machine out of stock
    }
}

Interview Q&A

State: States know about each other and manage transitions. State changes are internal — triggered by the state itself or the context based on conditions. The context's behavior changes automatically as state changes.
Strategy: Strategies don't know about each other. Client explicitly sets the strategy. No automatic transitions. Use State when you have a finite state machine; Strategy for interchangeable algorithms.

A state machine enum with transitions: use an enum for states and a Map<State, Map<Event, State>> for transitions. For complex state machines, use Spring Statemachine — a full state machine framework with guards, actions, transitions, hierarchical states. For simple cases, even a switch statement on an enum works — just less extensible.
