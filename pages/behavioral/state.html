20. State Pattern

Allows an object to alter its behavior when its internal state changes. The object will appear to change its class. Eliminates large conditional statements.

When to Use

  • When an object's behavior depends on its state and must change at runtime
  • When you have many conditional branches depending on the object's state
  • Order management systems, vending machines, traffic lights, ATMs
  • Spring Statemachine framework

Java Implementation

Order State Machine - State interface
// State interface
public interface OrderState {
    void confirm(Order order);
    void ship(Order order);
    void deliver(Order order);
    void cancel(Order order);
    String getStateName();
}
Order State Machine - Context
// Context
public class Order {
    private OrderState state;
    private final String orderId;

    public Order(String orderId) {
        this.orderId = orderId;
        this.state = new PendingState(); // initial state
        System.out.println("Order " + orderId + " created. Status: " + state.getStateName());
    }

    // Delegates all operations to current state
    public void confirm()  { state.confirm(this); }
    public void ship()     { state.ship(this); }
    public void deliver()  { state.deliver(this); }
    public void cancel()   { state.cancel(this); }

    public void setState(OrderState state) {
        System.out.println("[" + orderId + "] " + this.state.getStateName()
            + " → " + state.getStateName());
        this.state = state;
    }

    public String getStatus() { return state.getStateName(); }
}
Order State Machine - Concrete States
// Concrete States
public class PendingState implements OrderState {
    @Override
    public void confirm(Order order) {
        order.setState(new ConfirmedState());
    }
    @Override public void ship(Order order)    { System.out.println(" Cannot ship: order not confirmed"); }
    @Override public void deliver(Order order) { System.out.println(" Cannot deliver: order not shipped"); }
    @Override public void cancel(Order order)  { order.setState(new CancelledState()); }
    @Override public String getStateName()     { return "PENDING"; }
}

public class ConfirmedState implements OrderState {
    @Override public void confirm(Order order) { System.out.println(" Already confirmed"); }
    @Override
    public void ship(Order order) { order.setState(new ShippedState()); }
    @Override public void deliver(Order order) { System.out.println(" Cannot deliver: order not shipped"); }
    @Override public void cancel(Order order)  { order.setState(new CancelledState()); }
    @Override public String getStateName()     { return "CONFIRMED"; }
}

public class ShippedState implements OrderState {
    @Override public void confirm(Order order) { System.out.println(" Already confirmed & shipped"); }
    @Override public void ship(Order order)    { System.out.println(" Already shipped"); }
    @Override
    public void deliver(Order order) { order.setState(new DeliveredState()); }
    @Override public void cancel(Order order)  { System.out.println(" Cannot cancel: already shipped"); }
    @Override public String getStateName()     { return "SHIPPED"; }
}

public class DeliveredState implements OrderState {
    @Override public void confirm(Order order) { System.out.println(" Order already delivered"); }
    @Override public void ship(Order order)    { System.out.println(" Order already delivered"); }
    @Override public void deliver(Order order) { System.out.println(" Already delivered"); }
    @Override public void cancel(Order order)  { System.out.println(" Cannot cancel: already delivered"); }
    @Override public String getStateName()     { return "DELIVERED"; }
}

public class CancelledState implements OrderState {
    @Override public void confirm(Order order) { System.out.println(" Order is cancelled"); }
    @Override public void ship(Order order)    { System.out.println(" Order is cancelled"); }
    @Override public void deliver(Order order) { System.out.println(" Order is cancelled"); }
    @Override public void cancel(Order order)  { System.out.println(" Already cancelled"); }
    @Override public String getStateName()     { return "CANCELLED"; }
}
Order State Machine - Usage
// Usage
public class Main {
    public static void main(String[] args) {
        Order order = new Order("ORD-100");
        order.confirm();   // PENDING → CONFIRMED
        order.ship();      // CONFIRMED → SHIPPED
        order.cancel();    //  Cannot cancel: already shipped
        order.deliver();   // SHIPPED → DELIVERED
        order.cancel();    //  Cannot cancel: already delivered
    }
}

Interview Q&A

State: States know about each other and manage transitions. State changes are internal — triggered by the state itself or the context based on conditions. The context's behavior changes automatically as state changes.
Strategy: Strategies don't know about each other. Client explicitly sets the strategy. No automatic transitions. Use State when you have a finite state machine; Strategy for interchangeable algorithms.

A state machine enum with transitions: use an enum for states and a Map<State, Map<Event, State>> for transitions. For complex state machines, use Spring Statemachine — a full state machine framework with guards, actions, transitions, hierarchical states. For simple cases, even a switch statement on an enum works — just less extensible.
