Design Principles

Beyond SOLID and Design Patterns, these foundational principles (DRY, KISS, YAGNI, Law of Demeter, Composition) will drastically improve your code quality, maintainability, and architecture.

1. DRY (Don't Repeat Yourself)

Definition: "Every piece of knowledge must have a single, unambiguous, authoritative representation within a system."

Code duplication is the root of many software bugs. If logic is duplicated, a change in requirements means updating code in multiple places — and developers inevitably forget to update one of them.

DRY Violation & Fix
// BAD (WET: Write Everything Twice)
public void saveUser(User user) {
    if (user.getEmail() == null || !user.getEmail().contains("@")) {
        throw new IllegalArgumentException("Invalid email");
    }
    // save logic...
}
public void updateUser(User user) {
    if (user.getEmail() == null || !user.getEmail().contains("@")) {
        throw new IllegalArgumentException("Invalid email");
    }
    // update logic...
}

// GOOD (DRY)
private void validateEmail(String email) {
    if (email == null || !email.contains("@")) {
        throw new IllegalArgumentException("Invalid email");
    }
}
public void saveUser(User user) {
    validateEmail(user.getEmail());
    // save logic...
}
public void updateUser(User user) {
    validateEmail(user.getEmail());
    // update logic...
}

Warning: Beware of forced DRY. Sometimes two pieces of code look identical right now, but they represent entirely different business concepts that will evolve separately. Prematurely combining them tightly couples unrelated contexts code (violating SRP).

2. KISS (Keep It Simple, Stupid)

Definition: Most systems work best if they are kept simple rather than made complicated. Simplicity should be a key goal in design.

Do not use advanced design patterns, complex reflections, or massive generic structures when a simple if-else or standard loop solves the problem effectively and is easier for the next developer to read.

3. YAGNI (You Aren't Gonna Need It)

Definition: "Always implement things when you actually need them, never when you just foresee that you need them."

Writing code for features that might be needed in the future wastes time, introduces bugs, bloats the codebase, and usually results in the wrong architecture because the future requirements weren't fully understood yet.

  • Don't build a complex caching layer on day 1 if traffic is low.
  • Don't create abstract factories if you currently only have exactly one database type.
  • Focus on the requirements of the current sprint.

4. Law of Demeter (LoD)

Definition: The Principle of Least Knowledge. A module should not know about the inner details of the objects it manipulates. "Don't talk to strangers."

An object O should only invoke methods of:

  1. Itself
  2. Objects passed via method parameters
  3. Objects it instantiates
  4. Its direct dependencies/fields

It explicitly restricts "train wrecks" like: object.getChild().getGrandchild().doSomething().

Law of Demeter Example
// BAD (Train Wreck, violates LoD)
public class CustomerDashboard {
    public void printZipCode(Customer customer) {
        // Dashboard knows Customer has an Address, and Address has a ZipCode
        // This tightly couples Dashboard to Address!
        String zip = customer.getAddress().getZipCode();
        System.out.println(zip);
    }
}

// GOOD 
public class Customer {
    private Address address;
    // Customer delegates the call, hiding internal structure
    public String getZipCode() {
        return this.address.getZipCode();
    }
}

public class CustomerDashboard {
    public void printZipCode(Customer customer) {
        // Dashboard only talks to its direct friend, Customer
        String zip = customer.getZipCode();
        System.out.println(zip);
    }
}

5. Composition Over Inheritance

Definition: Classes should achieve polymorphic behavior and code reuse by their composition (containing instances of other classes) rather than inheritance from a base or parent class.

Inheritance introduces deep, rigid hierarchies (IS-A) that break encapsulation and are extremely difficult to change. Composition (HAS-A) allows assembling behavior dynamically at runtime and leads to flat, modular designs.

Inheritance vs Composition
// BAD: Inheritance leads to combinatorial explosions
class Animal {}
class FlyingAnimal extends Animal {}
class SwimmingAnimal extends Animal {}
class Bird extends FlyingAnimal {} 
class Duck extends Bird {} // Wait, a duck swims too! How do we inherit SwimmingAnimal?

// GOOD: Composition enables modular capabilities
interface FlyBehavior { void fly(); }
interface SwimBehavior { void swim(); }

class Wings implements FlyBehavior {
    public void fly() { System.out.println("Flapping wings"); }
}
class WebbedFeet implements SwimBehavior {
    public void swim() { System.out.println("Paddling"); }
}

class Duck {
    // Duck is COMPOSED with behaviors
    private FlyBehavior flying;
    private SwimBehavior swimming;

    public Duck(FlyBehavior flying, SwimBehavior swimming) {
        this.flying = flying;
        this.swimming = swimming;
    }

    public void move() {
        flying.fly();
        swimming.swim();
    }
}

6. Aggregation vs Composition

These are two types of associations in OOP describing how objects relate.

Property Composition Aggregation
Type "Part-of" (Strong relationship) "Has-a" (Weak relationship)
Lifecycle Child dies if Parent dies Child can exist independently of Parent
Real-World Example House & Room. If house is destroyed, rooms are destroyed. Department & Professor. If department closes, professor still exists.
