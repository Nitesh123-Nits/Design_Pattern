9. Decorator Pattern

Attaches additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.

When to Use

  • When you want to add behavior to individual objects without affecting others
  • When subclassing would lead to an explosion of classes
  • When you want to add/remove behaviors at runtime
  • Classic: Java IO streams, Spring Security filter chains, logging wrappers

Java Implementation

Component interface
// Component interface
public interface Coffee {
    String getDescription();
    double getCost();
}
Concrete Component
// Concrete Component
public class SimpleCoffee implements Coffee {
    @Override public String getDescription() { return "Simple Coffee"; }
    @Override public double getCost() { return 1.00; }
}
Abstract Decorator — wraps a Coffee and IS a Coffee
// Abstract Decorator — wraps a Coffee and IS a Coffee
public abstract class CoffeeDecorator implements Coffee {
    protected final Coffee coffee;  // wraps the component

    public CoffeeDecorator(Coffee coffee) {
        this.coffee = coffee;
    }

    @Override
    public String getDescription() { return coffee.getDescription(); }

    @Override
    public double getCost() { return coffee.getCost(); }
}
Concrete Decorators — each adds one feature
// Concrete Decorators — each adds one feature
public class MilkDecorator extends CoffeeDecorator {
    public MilkDecorator(Coffee coffee) { super(coffee); }
    @Override public String getDescription() { return coffee.getDescription() + ", Milk"; }
    @Override public double getCost()        { return coffee.getCost() + 0.25; }
}

public class SugarDecorator extends CoffeeDecorator {
    public SugarDecorator(Coffee coffee) { super(coffee); }
    @Override public String getDescription() { return coffee.getDescription() + ", Sugar"; }
    @Override public double getCost()        { return coffee.getCost() + 0.10; }
}

public class WhipDecorator extends CoffeeDecorator {
    public WhipDecorator(Coffee coffee) { super(coffee); }
    @Override public String getDescription() { return coffee.getDescription() + ", Whip Cream"; }
    @Override public double getCost()        { return coffee.getCost() + 0.50; }
}

public class VanillaSyrupDecorator extends CoffeeDecorator {
    private final int shots;
    public VanillaSyrupDecorator(Coffee coffee, int shots) {
        super(coffee);
        this.shots = shots;
    }
    @Override public String getDescription() { return coffee.getDescription() + ", Vanilla x" + shots; }
    @Override public double getCost()        { return coffee.getCost() + (0.30 * shots); }
}
Usage — compose at runtime
// Usage — compose at runtime
public class CoffeeShop {
    public static void main(String[] args) {
        // Order 1: Plain coffee
        Coffee coffee = new SimpleCoffee();
        System.out.println(coffee.getDescription() + " → $" + coffee.getCost());

        // Order 2: Coffee + Milk + Sugar
        Coffee latte = new SugarDecorator(new MilkDecorator(new SimpleCoffee()));
        System.out.println(latte.getDescription() + " → $" + latte.getCost());

        // Order 3: Double vanilla, milk, whip — built up step by step
        Coffee fancy = new SimpleCoffee();
        fancy = new MilkDecorator(fancy);
        fancy = new VanillaSyrupDecorator(fancy, 2);
        fancy = new WhipDecorator(fancy);
        System.out.println(fancy.getDescription() + " → $" + fancy.getCost());
        // Simple Coffee, Milk, Vanilla x2, Whip Cream → $2.35
    }
}

Java IO — Real-World Decorator in the JDK

Java IO Streams - Every IO wrapping is Decorator in action:
// Every IO wrapping is Decorator in action:
InputStream raw    = new FileInputStream("data.txt");             // component
InputStream buffed = new BufferedInputStream(raw);                 // decorator: adds buffering
InputStream gzipped = new GZIPInputStream(buffed);                // decorator: adds decompression
InputStreamReader reader = new InputStreamReader(gzipped, "UTF-8"); // decorator: bytes→chars
BufferedReader br = new BufferedReader(reader);                    // decorator: adds readLine()

Interview Q&A

Inheritance is static — decided at compile time. You'd need MilkSugarWhipCoffee, MilkVanillaCoffee, etc. — class explosion. Decorator is dynamic — wrap at runtime, compose any combination. Decorator follows the OCP: add behaviors by wrapping without modifying existing classes. Prefer Decorator when behaviors can be combined in many ways or toggled at runtime.

Spring AOP uses proxies (CGLIB or JDK dynamic proxies) that are essentially Decorators. When you use @Transactional, Spring wraps your bean in a proxy that adds transaction management logic (begin, commit, rollback) before/after your method — exactly like a Decorator. @Cacheable, @Async, @Retry all work the same way.

Decorator: Adds new behavior/responsibility. Client typically knows it's working with a decorated object and may wrap it themselves. Focus: feature addition.
Proxy: Controls access to the real object. Client believes it's working with the real object (same interface). Focus: access control, lazy loading, logging, caching. They look structurally similar but have different intents.
