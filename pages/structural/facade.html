🏢 Facade Pattern

Provides a simplified interface to a complex subsystem. Hides the complexity of the underlying system and provides a simple API for clients.

🎯 When to Use

  • When you want to provide a simple interface to a complex system
  • When there are many interdependent classes in a subsystem
  • When layering a system — Facade defines an entry point to each layer
  • Service layer in Spring (hides repo + business logic complexity from controllers)

💻 Java Implementation

E-Commerce Order Facade
// ── Complex Subsystem Classes ─────────────────────────────────

public class InventoryService {
    public boolean checkStock(String productId, int quantity) {
        System.out.println("Checking stock for " + productId + " × " + quantity);
        return true; // assume in stock
    }
    public void reserveStock(String productId, int quantity) {
        System.out.println("Reserving " + quantity + " units of " + productId);
    }
}

public class PaymentProcessor {
    public String processPayment(String userId, double amount) {
        System.out.println("Processing payment of $" + amount + " for user " + userId);
        return "TXN-" + System.currentTimeMillis();
    }
    public void refund(String transactionId) {
        System.out.println("Refunding transaction: " + transactionId);
    }
}

public class ShippingService {
    public String scheduleDelivery(String orderId, String address) {
        System.out.println("Scheduling delivery for " + orderId + " to " + address);
        return "SHIP-" + orderId;
    }
    public void cancelDelivery(String shipmentId) {
        System.out.println("Cancelling shipment: " + shipmentId);
    }
}

public class NotificationService {
    public void sendOrderConfirmation(String userId, String orderId) {
        System.out.println("Sending order confirmation email for order " + orderId);
    }
    public void sendShippingNotification(String userId, String trackingId) {
        System.out.println("Sending shipping notification, tracking: " + trackingId);
    }
}

public class LoyaltyPointsService {
    public void addPoints(String userId, double amount) {
        int points = (int)(amount * 10);
        System.out.println("Adding " + points + " loyalty points to " + userId);
    }
}

// ── Facade — One simple method replaces all subsystem calls ──
public class OrderFacade {
    private final InventoryService inventory;
    private final PaymentProcessor payment;
    private final ShippingService shipping;
    private final NotificationService notification;
    private final LoyaltyPointsService loyalty;

    public OrderFacade() {
        // Facade manages all subsystem instances
        this.inventory    = new InventoryService();
        this.payment      = new PaymentProcessor();
        this.shipping     = new ShippingService();
        this.notification = new NotificationService();
        this.loyalty      = new LoyaltyPointsService();
    }

    // ONE simple method wraps complex orchestration
    public String placeOrder(String userId, String productId, int qty, double price, String address) {
        System.out.println("\n=== Placing Order ===");

        // 1. Check stock
        if (!inventory.checkStock(productId, qty)) {
            throw new IllegalStateException("Out of stock: " + productId);
        }

        // 2. Reserve stock
        inventory.reserveStock(productId, qty);

        // 3. Process payment
        String transactionId = payment.processPayment(userId, price * qty);

        // 4. Generate order ID
        String orderId = "ORD-" + System.currentTimeMillis();

        // 5. Schedule delivery
        String trackingId = shipping.scheduleDelivery(orderId, address);

        // 6. Notify customer
        notification.sendOrderConfirmation(userId, orderId);
        notification.sendShippingNotification(userId, trackingId);

        // 7. Award loyalty points
        loyalty.addPoints(userId, price * qty);

        System.out.println("=== Order Complete: " + orderId + " ===\n");
        return orderId;
    }
}

// ── Client — only deals with Facade ──────────────────────────
public class CheckoutController {
    private final OrderFacade orderFacade = new OrderFacade();

    public void checkout(String userId, String productId, int qty, double price, String address) {
        // Client doesn't know about Inventory, Payment, Shipping — just one call!
        String orderId = orderFacade.placeOrder(userId, productId, qty, price, address);
        System.out.println("Order confirmed: " + orderId);
    }

    public static void main(String[] args) {
        new CheckoutController().checkout("user-123", "PROD-456", 2, 29.99, "123 Main St");
    }
}

🔍 Real-World Usage

  • Spring's JdbcTemplate — facade over JDBC's complex API
  • Spring's RestTemplate / WebClient — facade over HTTP
  • SLF4J — facade over logging frameworks (Log4j, Logback)
  • Service layer in Spring MVC — facade between controllers and repositories
  • Spring's ApplicationContext — facade over bean management

🎤 Interview Q&A

Facade: Simplifies interface to a complex subsystem for ease of use. Defines a new interface. Works with the subsystem as-is.
Adapter: Makes incompatible interfaces work together. Converts one interface to another that the client expects. Focuses on compatibility, not simplicity.

It can if taken too far. The Facade's responsibility is orchestration — coordinating the subsystem. As long as the Facade doesn't contain business logic (which lives in subsystem classes), it's fine. In Spring, the Service layer acts as a Facade but should delegate actual logic to repositories, domain objects, and specialized services.

SLF4J (Simple Logging Facade for Java) provides a unified logging API. Your code calls Logger.info(). At runtime, SLF4J delegates to whatever binding is on the classpath (Logback, Log4j2, java.util.logging). Your code doesn't change if you switch logging frameworks — the Facade absorbs the difference.
