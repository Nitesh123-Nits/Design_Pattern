Creational Patterns

Mechanisms for object instantiation while keeping clients decoupled from details.

①

Singleton

aka: Single Instance

Ensures a class has exactly one instance and provides a global access point to it. Created lazily on first use and reused subsequently.

When to use
  • Managing shared resources (DB pools, loggers, thread pools).
  • Centralized configuration management.
Why it helps

Prevents instance conflicts, reduces memory footprint, and provides a stable access point.

⚠ Overuse makes testing difficult. Prefer Dependency Injection where possible.
🏭

Factory Method

aka: Virtual Constructor

Defines an interface for creating an object but lets subclasses decide which class to instantiate.

When to use
  • Exact product type isn't known until runtime.
  • Frameworks where users must extend components (UI widgets).
Why it helps

Decouples client code from concrete classes. Adding new types doesn't break existing logic.

🏗️

Abstract Factory

aka: Kit

Creates families of related objects without specifying their concrete classes.

When to use
  • Family of objects must be used together (e.g., macOS buttons + menus).
  • Independence from concrete product representations.
Why it helps

Enforces consistency across a product family with a single point of failure (the factory).

🔨

Builder

aka: Step Builder

Separates complex object construction from its representation for step-by-step assembly.

When to use
  • Objects with many optional parameters (telescoping constructor problem).
  • Different representations for same data (HTML vs PDF).
Why it helps

Cleaner code, immutable objects, and avoids inconsistent intermediate states.

🧬

Prototype

aka: Clone

Creates new objects by copying an existing prototypical instance.

When to use
  • Costly initialization (DB fetches, complex math).
  • Avoid subclassing just to vary properties.
Why it helps

Faster than re-initialization. Useful for game spawning or template-heavy systems.

🏊

Object Pool

aka: Resource Pool

Maintains a set of pre-initialized objects ready for use instead of allocating on demand.

When to use
  • Costly creation (DB connections, HTTP clients).
  • High frequency of short-lived objects.
Why it helps

Reduces GC pressure and allocation latency. Core of connection pooling libs.

Structural Patterns

Combining classes and objects into larger, more flexible structures.

🔌

Adapter

aka: Wrapper

Converts one interface into another that clients expect. Bridges incompatible code.

When to use
  • Integrating 3rd-party libs with mismatching APIs.
  • Reusing legacy code without breaking changes.
Why it helps

Enables components to cooperate through a translation layer. isolates risk.

🌉

Bridge

aka: Handle/Body

Decouples abstraction from its implementation so both can vary independently.

When to use
  • Avoid permanent binding of abstraction/impl.
  • Orthogonal dimensions of inheritance (e.g., shapes x platforms).
Why it helps

Eliminates class explosion. New features don't touch existing implementation classes.

🌳

Composite

aka: Tree Structure

Composes objects into tree structures. Clients treat leaf and composite objects uniformly.

When to use
  • Tree structures (file systems, component trees, organizational charts).
  • Recursive operations on part-whole hierarchies.
Why it helps

Simplifies recursive processing. One call propagates through the entire structure automatically.

🎨

Decorator

aka: Wrapper

Attaches additional responsibilities to an object dynamically via composition.

When to use
  • Adding features without subclassing (logging, compression, caching).
  • Stacking behaviors selectively at runtime.
Why it helps

Flexible alternative to inheritance. Behaviors can be combined in any order.

🏢

Facade

aka: Simplified Interface

Provides a unified, simplified interface to a complex subsystem.

When to use
  • Hiding complexity of frameworks or multiple internal modules.
  • Layering systems to reduce inter-layer coupling.
Why it helps

Reduces cognitive load and coupling. Clients only see one clean interface.

🪶

Flyweight

aka: Cache

Uses sharing to support large numbers of fine-grained objects efficiently.

When to use
  • Huge populations of similar objects (game particles, text glyphs).
  • Memory constraints due to redundant object state.
Why it helps

Dramatically lowers memory usage by sharing immutable state across instances.

🔒

Proxy

aka: Surrogate

Provides a placeholder to control access to another object (protection, caching, lazy init).

When to use
  • Lazy loading expensive objects.
  • Access control or logging interceptions.
Why it helps

Interception layer is transparent to the client. Real object is only used when needed.

Behavioral Patterns

Managing object communication, distributed responsibilities, and algorithms.

⛓️

Chain of Responsibility

aka: Handler Chain

Passes a request along a chain of handlers. Each handler decides to process or forward.

When to use
  • Middleware pipelines (auth, logging, validation).
  • Event bubbling or nested permission checks.
Why it helps

Dynamic, decoupled processing. adding/removing handlers doesn't affect the sender.

📝

Command

aka: Action

Encapsulates a request as an object, allowing undo, queuing, and parameterization.

When to use
  • Undo/Redo history systems.
  • Asynchronous task execution or macro recording.
Why it helps

Requests are first-class citizens. They can be stored, replayed, or undone easily.

🔄

Iterator

aka: Cursor

Accesses elements of a collection sequentially without revealing its internal structure.

When to use
  • Uniform traversal across lists, trees, graphs.
  • Support multiple simultaneous traversals.
Why it helps

Abstracts storage logic. Clients write for interfaces, not implementation details.

📡

Mediator

aka: Controller Hub

Encapsulates interaction logic between a set of objects. Promotes loose coupling.

When to use
  • Complex UI forms or chat systems.
  • Tangled dependencies between component groups.
Why it helps

Converts M:N relationships into 1:N star topology. Components become simpler.

💾

Memento

aka: Snapshot

Captures object state without exposing internals, allowing for later restoration.

When to use
  • Checkpointing long tasks.
  • Save/Restore states in games or editors.
Why it helps

Preserves encapsulation. Originator manages its own state recovery.

👁️

Observer

aka: Pub-Sub

Notifies dependents automatically when a subject changes state.

When to use
  • Event listeners, reactive UIs, message buses.
  • One-to-many change propagation.
Why it helps

Decouples signal from action. foundation for all modern event-driven logic.

🔁

State

aka: FSM

Changes object behavior when its internal state changes. States are objects.

When to use
  • Vending machines, order workflows, traffic lights.
  • Replace complex switch/if-else chains.
Why it helps

Isolates state-specific logic. Adding new states is non-invasive.

♟️

Strategy

aka: Policy

Encapsulates algorithms into interchangeable objects. Clients decide logic at runtime.

When to use
  • Different sorting, compression, or payment methods.
  • Algorithm varies independently of context.
Why it helps

Enables Open/Closed Principle. Adding a strategy doesn't touch existing code.

📐

Template Method

aka: Skeleton

Defines algorithm skeleton in base class, deferring specific steps to subclasses.

When to use
  • Standardized workflows with custom hooks (parsers, report tools).
  • DRY for structural logic.
Why it helps

Inversion of Control (Hollywood Principle). Framework calls your overrides.

🚶

Visitor

aka: Double Dispatch

Adds operations to structures without modifying classes. Logic exists in visitors.

When to use
  • AST traversal, reporting on complex object hierarchies.
  • Add features to rigid structures.
Why it helps

Consolidates related logic. Elements don't need to know about all possible ops.

🚫

Null Object

aka: Default object

Substitute for null that provides do-nothing behavior to avoid null checks.

When to use
  • Optional collaborators without side-effects (Stub logger).
  • Remove repetitive null-guard noise.
Why it helps

Cleaner client code. Logic flows smoothly without interruption marks.

Architectural Patterns

High-level system structure and cross-service communication patterns.

🏗️

MVC

Model-View-Controller

Separates application into data logic (Model), presentation (View), and input (Controller).

When to use
  • Web & desktop UIs where data/UI change independently.
Why it helps

Decouples business rules from display technology. Parallel development.

⚡

CQRS

Command Query Segregation

Splits read (Query) and write (Command) logic into separate models for performance.

When to use
  • High-scale systems where read/write demands differ greatly.
  • Complex domains requiring denormalized read views.
Why it helps

Optimize read/write paths independently. Scalability at the architectural level.

🏹

Saga

Distributed Transactions

Manages long-running distributed steps via local transactions and compensation logic.

When to use
  • Microservices where 2-phase commit is too slow/risky.
  • Event-driven workflows spanning multiple services.
Why it helps

Eventual consistency without global locks. Graceful failure recovery.

🔌

Circuit Breaker

aka: Fail-Fast Gate

Wraps service calls to monitor failures and open the circuit to prevent cascade crashes.

When to use
  • Integration with fragile 3rd-party APIs.
  • Inter-service communication in distributed systems.
Why it helps

Fault tolerance. System degrades gracefully instead of hanging on timeouts.
